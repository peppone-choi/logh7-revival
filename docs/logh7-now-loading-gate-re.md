# NOW LOADING dismiss 게이트 정적 RE

**대상:** `g7mtclient.exe` (ImageBase 0x400000, ASLR off)
**분석 소스:** Ghidra headless 디컴파일 (`scratchpad/ghidra-out/g7mtclient-sjis.exe_decompiled.c`, 11593 함수)
**목적:** 월드 진입(gamemode=0, ring=1) 후 objTable·그리드셀·캐릭터가 다 찼는데도 "NOW LOADING"(loadingbg00.jpg)이 안 걷히고 전략맵으로 전환 안 되는 원인을 정적으로 확정.

## 결론 (첫 줄)

**NOW LOADING dismiss 조건 = 로딩 페이드 float `clientBase+0x357e88` 이 1.0 에 도달하는 것. 이 페이드는 월드-로드 파이프라인(FUN_004b68f0)의 정적정보 워크(FUN_004b76e0)가 완주해야 램프되는데, 워크의 각 스텝은 "대기 응답 큐(`clientBase+0x357ec0`) == 0" 일 때만 다음 요청을 보낸다. 즉 클라는 이미 보낸 요청의 응답을 서버가 정확한 메시지 코드로 돌려줄 때까지 로딩을 물고 멈춘다.**

라이브 관측("클라가 0x0304/0x0306/0x0314 요청 후 정지")과 코드가 정확히 일치한다: 워크 step0 이 0x0304(→응답 0x0305)와 0x0306(→0x0307)을, step1 이 0x0314(→0x0315)를 보낸다. 세 요청이 전선에 나갔다는 것은 **0x0305·0x0307 은 도착(팝됨)했고, 마지막 0x0314 의 응답 0x0315 가 안 왔다**는 뜻이다.

**→ 1차 서버 수정: 월드 서버가 정적정보 요청 각각에 대해 정확히 매칭되는 응답 코드를 돌려줘야 한다. 당장의 블로커는 `0x0314 → 0x0315` 응답 누락.** 이후 워크가 보내는 나머지 요청(0x0312/0x030a/0x0310/0x030e/0x031c/0x0308/0x030c/0x0300/0x0f00/0x0f02)도 전부 응답해야 완주한다. (전체 표는 §5.)

**objTable 가 이미 찬 것은 이 게이트와 무관하다** — objTable 는 그리드-엔터 푸시(0x0b0a NotifyEnterGridEnd)가 `FUN_004c2a80(1)` 을 호출해 채운 것이지(§6), 로딩 워크 완주(FUN_004c2a80(0), 워크 마지막 스텝)로 채운 게 아니다. 두 경로는 독립이다.

## 7. 서버 수정 지시 (구체)

### ★1차 (확신도 高) — 정적정보 응답 전부 구현
월드 서버가 정적정보 워크의 **모든 요청에 정확한 응답 코드로 답**해야 한다. 현재 블로커는 `0x0314→0x0315`. 순서대로 필요한 응답(§5 워크 표):
```
0x0314→0x0315, 0x0312→0x0313, 0x030a→0x030b, 0x0310→0x0311, 0x030e→0x030f,
0x031c→0x031d, 0x0308→0x0309, 0x030c→0x030d, 0x0300→0x0301,
0x0f00→0x0f01, 0x0f02→0x0f03
```
(이미 동작하는 0x0304→0x0305, 0x0306→0x0307 은 유지.) 각 응답은:
1. **정확한 메시지 코드**(위 표) — 다른 코드면 클라 큐 매칭 실패 → 무시 → stuck.
2. **길이가 클라 사이즈 테이블과 일치** — 안 맞으면 recv 프레이밍이 깨져 그 코드로 전달 안 됨.
3. payload 내용은 무엇이든(0 채움 가능) 게이트 통과엔 무관 — 핸들러가 owner-id 를 -1 로 고정하므로. 렌더 정합은 별도 문제.

### ★2차 (브리프 가설 검증) — 0x0f00/0x0f02 는 지금 블로커 아님
0x0f00 은 워크 step10, 0x0f02 는 step0xf 에서 전송된다. 클라는 step1(0x0315 대기)에서 멈춰 **step10/0xf 에 도달하지 못하므로 0x0f00/0x0f02 를 아직 안 보낸다.** 서버가 0x0314 시점에 0x0f03 을 조기전송해도, 그때 큐 헤드는 0x0315 를 기대하므로 0x0f03 은 매칭 실패로 **무시**된다(해롭진 않음). 단 그 결과 나중에 클라가 0x0f02 를 보낼 때 서버가 0x0f03 을 **다시** 보내야 한다(1회성 조기전송이면 step0xf 에서 stuck). → 조기전송 금지, 반드시 요청-응답 페어로.

### 검증 방법 (라이브 재캡처)
서버 수정 후 클라 메모리를 읽어 확인 — **어떤 응답을 기다리는지 직접 지목 가능:**
- `clientBase+0x357ec0` (대기 카운트) → 0 이 되어야 워크 진행.
- `clientBase+0x357ec4` (u16, 대기 큐 헤드 send 코드) / `clientBase+0x357ec8` (기대응답 코드) → **지금 무슨 응답을 기다리는지.**

# 후속 RE: 0x0315 만 워크 큐 미팝 (Frida 실측 후)

**실측(Frida):** `walkStep=0x2, waitCount=1, queue=[{send:0x314, exp:0x315}]`. 서버 0x0315 = 코드·크기(5004B) 정확. 그런데 0x0304/0x0306 은 팝됐고 0x0315 만 미팝.

## 결론 (첫 줄)
**0x0315 팝 조건 = 디스패처 FUN_004ba2b0 이 param_1==0x315 로 호출되는 것뿐(핸들러가 local_1c=-1 고정 → 코드일치만이 관건). 0x0305 와 팝 경로가 100% 동일하므로, 미팝은 팝 조건 문제가 아니라 "0x0315 프레임이 코드 0x315 로 디스패치되지 못함"(수신/프레이밍 desync) 이다.**

## 1. 0x0305 vs 0x0315 핸들러 정적 대조 (팝 경로 동일)
두 핸들러 모두 같은 스위치(switch @ 38199) 안에 있고, `break` 후 **동일하게 line 38774 `goto LAB_004bdd33`** 로 빠져 팝 tail 로 간다.

| | 0x0305 (case @38200) | 0x0315 (case @38379) |
|---|---|---|
| 로그 | ResponseStaticInformationCard | ResponseStaticInformationGrid |
| payload 복사 | 0x1482 dword+2B → clientBase+0x3e0c8c | 0x4e3 dword(=5004B) → clientBase+0x3f4448 |
| **local_1c** | **= -1** (line 38209) | **= -1** (line 38388) |
| 추가 처리 | 없음 (tail 2바이트 복사만) | **`FUN_004abbb0(clientBase+0x3f444c, param_2)` RLE 디코드** |
| 종료 | break → LAB_004bdd33 | break → LAB_004bdd33 |

- 유일한 차이는 **FUN_004abbb0(RLE 그리드 디코더)** 호출. 그러나 이 함수는 (a) 예외 안 던짐(경계검사 line 26477 `if bVar1*bVar2 < uVar9 goto 종료`), (b) 쓰기 대상은 clientBase+0x3f444c 전역(스택·pending 큐 0x357ec0 건드리지 않음), (c) 반환값을 핸들러가 무시. **local_1c=-1 세팅 이후·break 이전에 호출되므로 팝을 막을 수 없다.**
- ∴ **0x0315 가 디스패치만 되면 반드시 팝된다.** 팝 조건(owner-id)은 0x0305 와 동일하게 통과.

## 2. 팝 tail 매칭 조건 (재확인, FUN_004ba2b0 line 41058~)

## 4. 서버 수정 지시
1. **모든 정적정보 응답 payload 길이를 위 테이블값과 바이트 단위로 정확히 일치**시켜라(패딩/절단 금지). 프레임 헤더 포맷(코드+길이필드 유무)은 이미 팝된 0x0305/0x0307 과 동일해야 한다.
2. **최우선 감사 대상: 0x0307(정확히 58802B) 와 0x0315(정확히 5004B).** 0x0307 이 한 바이트라도 어긋나면 0x0315 가 깨진다.
3. 0x0315 payload = RLE 그리드. 디코드 유효성(ΣrunLen==width×height)은 팝과 무관하니 지금은 길이만 맞으면 된다. 단 길이는 반드시 5004B.

## 5. 결정적 다음 프로브 (팝 문제 vs 프레이밍 문제 확정)
**FUN_004ba2b0 @ 0x004ba2b0 진입점을 Frida 훅해 param_1(디스패치 코드) 전량 로깅:**
- 0x315 가 **디스패치되면(param_1==0x315) 정적상 무조건 팝** → 만약 그래도 미팝이면 self_id(clientBase+0x3584a0) 및 local_1c 실측(정적 예측 위반, 재조사).
- 0x315 가 **한 번도 디스패치 안 되면 → 수신/프레이밍 문제 확정**(§3). 이어서 수신 큐(clientBase+0x3552b8, stride 0x14, 코드 @+0x3552bc / 크기 @+0x3552c4)를 덤프해 0x0307 직후 어떤 코드·크기로 밀렸는지 확인. 또는 서버 송신 바이트를 캡처해 0x0305/0x0307 실제 길이를 테이블(20994/58802)과 대조.

## 확정 / 미확정
**확정:** 0x0315 핸들러 팝 경로 = 0x0305 와 동일(local_1c=-1, 동일 goto LAB_004bdd33). FUN_004abbb0 는 팝 방해 불가(경계검사·전역쓰기·반환무시). 미팝의 유일한 정적 원인 = param_1≠0x315(디스패치 안 됨). 클라는 코드별 고정크기 테이블로 프레이밍(길이필드 아님) → 앞선 메시지 길이 불일치가 desync 유발.
