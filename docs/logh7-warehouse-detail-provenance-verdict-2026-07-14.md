> **결합할 정본(canonical) 재고 값은 존재하지 않는다.** `0x0327` 창고 레코드의 **레이아웃과 각 필드의 게임 의미는 정본(P1, 라이브 확정)** 이지만, 각 base의 창고가 담는 **실제 수량**(함정 재고·병력 재고·軍需物資/食料/資源 스칼라)을 기록한 정본 원천은 매뉴얼·星系図·MsgDat·EXE·CD 어디에도 없고 원본 서버 시드는 소실됐다. 창고 재고는 원본이 런타임에 자동생산 누적·할당 소비로 계산하던 **런타임 상태**이며, 채워 넣을 수 있는 값은 절차생성(P3)뿐 정본이 아니다. → **판정 (b) 런타임 상태 / 실무 귀결 (c) 빈 창고**. 경제(`[[logh7-economy-unimplemented-canon-verdict-2026-07-14]]`)와 결론은 같으나 논리는 미묘하게 다르다(아래 §4).

---

## 1. `0x0327` 창고 레코드가 담는 것 (의미 확정, P1)

`0x0326 RequestInformationWarehouse` 는 `{base:u32BE, outfit:u32BE}` 8바이트로 특정 base+outfit의 창고를 요청하고, `0x0327` 이 그 base의 **惑星倉庫(planet warehouse)** 내용을 고정 0x300바이트로 응답한다. renderer `FUN_0057aa90`가 캐시(`clientBase+0x3e098c`)에서 읽는 필드 대응:

| wire 필드 (`warehouse-record.mjs`) | 캐시 오프셋 | 게임 개념 | 근거 |
|---|---|---|---|
| `ships[]` = kind u16 + unitNumber u8 + boatNumber u16 | `+0xC` u8 개수 / `+0x10` stride6 `+0` u8 수량 | 창고에 보관된 **艦艇 유닛** (함종별 재고·정수·정수) | 매뉴얼 §8.2 "완성 생산이 惑星倉庫에 자동 입고"; 施設 `係留所`(艦艇 보관) `倉庫`(자원/軍需/食料 보관) |
| `troops[]` = kind u16(tag) + troopGrade u8 + unitNumber u16 | `+0x260` u8 개수 / `+0x262` stride6 tag+값 | 창고에 보관된 **乗組員/陸戦兵 유닛** (분류 tag별 수량·수련도) | 매뉴얼 §8.5 補充(乗組員·陸戦兵 유닛); 施設 `兵舎` |
| `supplies` u32 | `+0x2F4` u32 | **軍需物資** 재고량 | `place-facilities.json` `倉庫` = "資源／**軍需物資**／食料を保管する施設" |
| `food` u32 | `+0x2F8` u32 | **食料** 재고량 | 同上 (農業プラント 생산 → 倉庫 보관) |
| `mineral` u32 | `+0x2FC` u32 | **資源** 재고량 | 同上 (資源採掘施設 채굴 → 倉庫 보관) |
| `base`/`outfit`/`index` u32 | `+0x0`/`+0x4`/`+0x8` | 요청 echo 식별자 (창고 소유 base/부대/인덱스) | 요청 body echo, run7 라이브 base=70 확정 |

**의미 확정 신뢰도 P1**: `倉庫` 施設 정의("資源/軍需物資/食料 보관")가 스칼라 3필드와 정확히 일치하고, 매뉴얼 §8.2 창고 개념(艦艇+乗組員/陸戦兵+軍需物資 입고)이 배열 2종과 일치한다. 오프셋·엔디안·개수 규약은 라이브 grade-a로 별도 확정(배경).

### `0x031f` ResponseInformationBase 와의 관계

**별개 메시지, 다른 뷰.** `0x031f`는 base 단위 집계 배열(`commodity[3]`, `transport_supplies[30]`, `outfit_supplies[30]`, `budget[5]`, `budgeting[6]` + 스칼라, `[[logh7-info-records-wire.md]]` HIGH)을 싣고, `0x0327`은 base+outfit의 **창고를 유닛 단위로 품목화**해 싣는다. 경제 인접 필드(`transport_supplies`/`outfit_supplies`)가 개념상 가장 가깝지만, 둘 다 값의 정본은 없다. `0x0327`은 `0x031f`의 하위 집합이 아니라 창고 상세 전용 응답이다.

## 2. 필드별 정본 원천 유무 (핵심 판정)

| 필드 | 레이아웃 정본 | **값(수량) 정본** | 등급 | 근거 경로 |
|---|---|---|---|---|
| 재고 함정 배열 `ships[]` | 있음 (P1, 라이브) | **없음** | — | 함종 enum은 kind-id 카탈로그(P1)에 있으나, **base별 창고 재고 수량 테이블은 부재.** `auto-production.json`은 "무엇을 생산하는가"(품목 리스트)이지 "얼마나 쌓여 있는가"가 아님 |
| 재고 병력 배열 `troops[]` | 있음 (P1, 라이브) | **없음** | — | 同上. 乗組員/陸戦兵 품목은 `auto-production.json`(P1)에 있으나 재고 수량은 없음 |
| 軍需物資 스칼라 `supplies` | 있음 (P1, 라이브) | **없음** | — | `content-recovery-economy §3 absent` 전수검색 0건: 매뉴얼·星系図·MsgDat 9,582레코드·EXE 어디에도 base별 補給/軍需 수치 없음 |
| 食料 스칼라 `food` | 있음 (P1, 라이브) | **없음** | — | 同上. `食料` 는 경제 미구현 판정의 스칼라 집합에 포함 |
| 資源 스칼라 `mineral` | 있음 (P1, 라이브) | **없음** | — | 同上 |

**전수 확인 결과 (조사 지점 2):**
- `server/content/manual/logistics-economy.json`(매뉴얼 pp.41-45): 창고·할당·재편성·보충의 **규칙**만 정의. base별 초기 재고표 없음.
- `server/content/auto-production.json`(매뉴얼 pp.76-78, P1): 자기문서화 `_note` 명시 — **"Counts = number of production entries listed. The three columns are not row-paired tuples."** 즉 "행성이 어떤 품목을 자동생산하는가"의 리스트이지 창고 재고 수량이 아니다. 이를 재고 수량으로 제시하면 날조.
- `server/content/initial-deployment.json`(매뉴얼 p75, P1): 部隊初期配置 = **필드에 배치된 부대**(사령관에 할당된 함대/초계/지상)이지 창고 보관품이 아니다.
- `server/content/manual/place-facilities.json`: `倉庫` 施設 = 저장 시설의 **존재**만 정의, 초기 저장량 없음.
- `server/data/seed/*.json`: 창고/재고/stock 파일 자체가 없음(factions/characters/fortresses/galaxy/ships/rank/deployment만).
- `docs/reference/legacy-evidence/logh7-content-recovery-economy-2026-06-29.md` §3 absent: **행성별 補給/장갑/자원 등 스탯의 캐논 값은 어떤 소스에도 없음**, 원본 서버 데이터("original server data is gone") 소실 명시. 창고 재고도 이 부재에 포함된다.

## 3. 결합 가능한 것 vs. 없는 것

**결합할 정본 값은 없다.** 채우려면 절차생성이 유일 수단이고 그것은 P3 — 정본이 아니다.

특히 유혹적인 오용 하나를 명시적으로 배제한다: `auto-production.json`의 품목 개수를 창고 재고 수량으로 매핑하는 것. 이 파일의 `_note`가 "Counts = number of production **entries**"라고 스스로 밝히므로, 그 값을 "창고에 함정 N척 있음"으로 제시하면 **원본에 없던 콘텐츠를 지어내는 것**이다. 하지 말 것.

결합 가능한 것은 **값이 아니라 enum/의미**뿐이다(그리고 그것도 재고를 만들어내지 않는다):
- `ships[].kind` 를 채운다면 kind-id 정본 카탈로그(최근 커밋 `9d8661be` 유닛 kind-id enum)를 써야 한다 — 그러나 이는 "어떤 함종 슬롯인가"의 라벨 정본이지 "몇 척 재고"의 값 정본이 아니다.
- `troops[].kind`(tag)도 마찬가지. QA 마커의 tag 0x10/0x11은 임의값(positive control)이며 정본 매핑이 아니다.

## 4. (a)/(b)/(c) 판정과 경제 판정과의 차이

**주 판정: (b) 런타임 상태.** 창고 재고는 원본이 자동생산 누적(입고)과 할당·보충(출고)으로 매 틱 계산하던 상태다. 시나리오 시작(801-07) 시점의 값도 원본 서버가 시드했을 뿐, 규칙에서 파생되는 런타임 값이지 정본 테이블이 아니다. 그 원본 시드는 소실됐다.

**실무 귀결: (c) 빈 창고(전부 0)가 정직한 기본.** 정본 값이 없으므로 서버는 base ID만 echo하고 재고·스칼라를 0으로 두는 현재 보수 계약(`warehouse-record.mjs` 기본 경로, 테스트 "게이트 off면 base 외 전부 0")이 정직하다. 절차값을 정본으로 승격하지 않는 한, 이 이상 채울 근거가 없다.

**(a) 정적 정본은 아니다**: base별 초기 재고표가 매뉴얼/데이터 어디에도 없음(§2 전수 확인).

### 경제 판정과의 미묘한 차이 (정직성 주의)

- **경제**: 원본이 기능 자체를 **미구현**(`経済関連は現在未実装`)으로 출시 → 0이 **적극적으로 옳다**(원본 화면도 비어 있음).
- **창고**: 기능은 **정본 구현**이다(자동생산이 입고하고 할당/보충이 소비하는 정식 물류 루프, 매뉴얼 §8 전체). 원본은 실제 재고를 갖고 돌아갔다. 다만 그 **초기 재고 값의 정본이 (i) 어떤 추출 소스에도 기록된 적 없고 (ii) 원본 서버 시드가 소실**되어, 지금 채울 수 있는 것이 P3 절차값밖에 없을 뿐이다.

즉 "빈 창고 = 정본 동작"이라는 표현은 경제만큼 강하게 쓰면 안 된다. 정확히는 **"창고는 정본 기능이나 재고 값의 정본이 부재/소실 → 빈 창고가 현재 정직한 기본이며, 미래에 자체 물류 sim이 자동생산 규칙(P1)으로 재고를 계산해 채우는 것이 정당한 경로(그 산출값은 P3로 태깅)"** 이다. 자동생산·할당·보충 **규칙**은 P1 정본이므로, 재고를 사후에 시뮬레이션으로 생성하는 것은 정당하되 그 결과값을 정본이라 부르지 말 것.

## 5. 이 판정이 확정하는 것

1. `0x0327` 상세 렌더 병목의 남은 문제는 "값의 정본 원천 찾기"가 아니다 — **그런 원천은 없다.** 배선(오프셋·엔디안·tag)은 라이브로 열렸고, 값은 (a) 빈 창고로 두거나 (b) 자동생산 규칙(P1) 기반 자체 물류 sim으로 P3 재고를 산출하는 두 경로만 남는다. 정본 재고 결합이라는 작업 항목은 **닫는다(closed — no-canon values).**
2. `auto-production.json` 품목 개수를 창고 재고로 오용하지 말 것(§3).
3. 향후 재고를 채울 때: 자동생산 규칙(P1)으로 계산하되 산출 데이터에 `_grade: P3` / 절차생성 헤더를 붙이고, 어떤 보고에서도 정본으로 표기 금지(경제 `planet-economy.json` provenance 규칙과 동일).

## 결론 (1문장)

**`0x0327` 창고 레코드의 레이아웃과 각 필드의 게임 의미는 정본(P1)으로 확정되지만 base별 재고 값을 담은 정본 원천은 어디에도 없고 원본 서버 시드는 소실됐으므로, 결합할 정본 값은 존재하지 않으며(판정 b 런타임 상태 / 귀결 c 빈 창고), 재고를 채운다면 자동생산 규칙(P1)에서 파생한 P3 절차값으로만 가능하고 그것을 정본이라 불러선 안 된다.**
