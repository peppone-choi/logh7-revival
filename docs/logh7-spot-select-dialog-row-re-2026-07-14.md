# 寄港 拠点 SelectDialog 리스트 행 좌표 정적 RE (2026-07-14)

## 요약 (결론 먼저)

**행의 화면 rect는 메모리에 저장되지 않는다.** 리스트 위젯(`parent+0x244`)과 그 안의
그리드(`list+0x20`)는 **논리 모델**(행 연결리스트 + 셀 상태 플래그 배열)만 들고 있고,
화면 좌표는 **그리기/히트테스트 시점에 그리드 입력엔진 `FUN_005015f0` 계열이 전역 입력·레이아웃
상태(`0x022142a8`/`0x022143e4` 등)로부터 그때그때 계산**한다. 따라서 정적 메모리에서
"원점 + 행높이 × 인덱스"에 넣을 stored 스칼라를 찾는 접근은 이 위젯 클래스에 대해 **구조적으로
불가능**하다. 트레이서가 시도한 5개 origin 후보가 전부 실패한 것은 계산 버그가 아니라
**입력 대상 객체가 위치 위젯이 아니기 때문**이다.

행 N을 클릭하려면 정적 좌표 계산이 아니라 다음 중 하나의 **동적** 경로를 써야 한다:
- (A) 그리드 히트테스트/렌더 함수 `FUN_005015f0`(경계 훅)의 out-struct에서 엔진이 방금 계산한
  셀 rect를 읽어 캡처.
- (B) 픽셀 클릭을 우회해 인덱스로 선택: 리스트 메서드 `FUN_00576d40(index)`가 선택을 바꾼다
  (ECX=list). 이후 決定 트리거.

아래 각 질문에 grade와 정본 EXE 실바이트 검증을 붙인다.

## 검증한 정본 EXE 실바이트 (grade a 근거)

Ghidra 디컴파일은 `-sjis` 사본 기반이라 레지스터 할당이 다르다(예: `this`가 ECX 대신 ESI).
**오프셋과 상수·의미는 정본 EXE(sha256 9c97…bb51)에서 동일함을 바이트로 확인.**

| VA | 정본 파일오프셋 | 실바이트(선두) | 해석 |
|---|---|---|---|
| `0x004fa350` | `0xfa350` | `8a 44 24 04 53 33 db 56 8b f1 c7 46 04 88 13 00 00 c7 46 08 19 00 00 00` | list 위젯 ctor. `mov [esi+4],0x1388(5000)` · `mov [esi+8],0x19(25)` |
| `0x00507090` | `0x107090` | `83 ec 08 56 8b f1 8b 46 08 3b 06 75 0f 6a 00 68 40 a2 77 00 e8 …` | origin walker. `mov eax,[esi+8]` · `cmp eax,[esi]` (this+8 vs this+0) |
| `0x00577050` | `0x177050` | `64 a1 00 00 00 00 6a ff 68 d8 6a 66 00 …` | list 행추가(SEH 프레임) |
| `0x005030f0` | `0x1030f0` | `53 56 57 8b f9 8b 4c 24 10 8a 41 08 84 c0 …` | 그리드 셀 플래그 setter |
| `0x00582060` | `0x182060` | `6a ff 68 83 73 66 00 64 a1 00 00 00 00 …` | factory 0x2d handler(기존 확정) |

`mov [esi+4],0x1388` / `mov [esi+8],0x19` 두 연속 명령이 정본 바이트에 그대로 존재 →
**+0x4=5000, +0x8=25는 ctor 리터럴 상수**임을 확정.

## Q1. 리스트 위젯 객체의 실제 구조

**답(grade a):** `0x00ca3954`는 이 다이얼로그의 리스트 위젯이 맞다. 단, 그것은 **정보패널 정적
전역 객체 `0x00ca3710`에 인라인 임베드된 서브객체**이며 `list = panel + 0x244`이다
(`0xca3954 - 0xca3710 = 0x244`, 트레이서가 런타임에 쓰는 `parent+0x244`와 일치).
트레이서가 정적 `0x00ca3954`에서 `itemCount=1`·`rowBaseIds=[70]`을 이미 성공적으로 읽었다는
사실이 이 주소가 라이브 객체임을 뒷받침한다(패널이 힙객체라면 정적 읽기가 쓰레기였을 것).

**리스트 위젯 필드맵** (ctor `FUN_004fa350` + 메서드 `FUN_00577050`/`FUN_00576d40`/`FUN_00576ec0`
교차확인, grade a):

| 오프셋 | 값/의미 | 근거 |
|---|---|---|
| `+0x00` | 위젯 id (param_1, 또는 `DAT_02222554+0x34` 자동발번) | ctor `*in_ECX=param_1` |
| `+0x04` | **상수 5000 (0x1388)** — 화면좌표 아님 | ctor `in_ECX[1]=5000` |
| `+0x08` | **상수 25 (0x19)** — 부모 위젯 핸들 슬롯/기본값, 행높이 아님 | ctor `in_ECX[2]=0x19`; origin walker가 부모링크로 사용 |
| `+0x0c` | **컬럼 수 = 1** (단일 컬럼 리스트) | `FUN_00577050`이 셀 루프 카운트로 사용 |
| `+0x10` | 현재 선택 수 (멀티셀렉트 카운터) | `FUN_00576d40` |
| `+0x14` | 최대 선택 수 = 1 (단일 선택) | ctor `in_ECX[5]=1`, `FUN_00576d40` |
| `+0x20` | **그리드 객체 포인터** (셀 상태 모델) | `FUN_00577050`/`FUN_005030f0` |
| `+0x8d4` | 선택 모드 | `FUN_00576d40` |
| `+0x8d8` | 선택 FSM 상태(2↔3 토글) | `FUN_00576d40`/`FUN_00576c60` |
| `+0x8e0` | **std::list sentinel 노드** (행 연결리스트 head) | ctor `in_ECX[0x238]=FUN_00648d42(0x20)` |
| `+0x8e4` | 행 개수(itemCount) | ctor=0, `FUN_00577050` 증가 |
| `+0x8e8` | 선택 인덱스 | `FUN_00576d40` |
| `+0x8ec` | 직전 선택 인덱스 | `FUN_00576c60` |

**행 배열 위치·stride(grade a):** 행은 배열이 아니라 **이중연결 std::list**다. sentinel=`list+0x8e0`,
각 노드 페이로드=`node+8`, **baseId=payload+8 (즉 node+0x10)**. 순회는 `FUN_00576ec0`이 정본
(`node = *node`로 전진, sentinel 만나면 종료). 트레이서의 행 순회(`node.add(8)`, `baseIdAt08`)와
일치.

## Q2. 행의 화면 좌표를 결정하는 것

**답(grade a): 행 rect는 메모리에 저장되지 않는다. 그리기/히트테스트 시점에 계산된다.**

`list+0x20` 그리드 객체는 셀의 **상태 플래그 바이트**만 저장한다. 그리드 init(`0x5064xx` 계열,
`FUN_005030f0`)이 컬럼당 블록(stride `0x3f50`)을 잡고 `+0x90d0`에 행별 1바이트 플래그(0/1/2 =
일반/선택/모서리), `+0x9330`에 행 수(600 고정), `+0x9334`에 행별 3-dword 논리 레코드를 두는데
**이 중 어느 것도 픽셀 rect가 아니다**(전부 0으로 초기화되는 모델 데이터).

실제 좌표는 입력/렌더 엔진 `FUN_005015f0`(opcode 디스패처)이 계산한다. 이 함수는 전역
마우스/좌표 상태 `DAT_022142a8`(=트레이서 `INPUT_STATE 0x022142a8`), `DAT_022142ac/d0/d4`,
`DAT_022143e4/e8`를 읽어 out-struct(`param_3`)에 rect류 값을 채운다:
`param_3[0..1]`(=index/id류), `param_3[2..3]`(좌표), `param_3[4..5]`(마우스 좌표). 리스트는
`FUN_00576b30`에서 `FUN_005015f0(0xe, grid, out28, 0)`로 이 엔진을 호출한다.

즉 **계산식·입력값이 정적 상수가 아니라 런타임 레이아웃/입력 전역에 있다.** "원점+행높이×N"
형태의 stored 소스는 존재하지 않는다.

## Q3. 다이얼로그의 절대 화면 원점 / 5개 후보가 실패한 이유

**답(grade a, 실패 원인 확정 / grade c, 올바른 원점 소스 미확정):**

`FUN_00507090`(트레이서의 `READ_ABSOLUTE_ORIGIN`)은 **위치 위젯 전용** 원점 walker다:
- this+0x8 = 부모 위젯 핸들 → `FUN_00502760`→`FUN_0050cf40`로 부모 객체 해석
- this+0x0 = 자기 id (assert: `if(this+8==this+0)`)
- **this+0xC = local x, this+0x10 = local y**
- 절대원점 = 부모절대원점 + (this+0xC, this+0x10) 재귀 누적

5개 후보가 전부 실패한 이유:
1. `listStatic`/`listDeref` → **null**: 리스트 위젯의 `+0x0`은 vtable이 아니라 **정수 id**라
   트레이서의 vtable 가드(`readPtr(list) >= 0x10000`)가 걸러냈다. 설령 호출해도 리스트의
   `+0xC/+0x10`은 **컬럼수(1)/선택수**라서 x=1 같은 무의미 값이 나온다.
2. `parentStatic`/`parentDeref` → **쓰레기 좌표**: 패널 모델 객체의 `+0xC/+0x10`도 위치필드가
   아니다(패널은 `+0x234` panelKind, `+0x244` list 등을 갖는 복합 모델). `FUN_00507090`이
   비좌표 필드를 x/y로 읽어 포인터성 쓰레기(116382976 등)를 반환.
3. `column0` → **null**: `readPtr(list+0x20)`은 셀 위젯이 아니라 **그리드 모델 포인터**다.

**결론:** 이 리스트를 그리는 "위치 위젯"은 리스트/패널 모델 객체와 별개이며, 그 원점은 정적
메모리의 한 필드로 노출되지 않는다(다이얼로그 프레임 레이아웃이 draw time에 결정). 어느 위젯이
호스트인지 **정적으로 확정하지 못했다** — 이것을 확정하려면 라이브에서 (a) `FUN_005015f0`
경계 훅으로 out-struct 좌표를 캡처하거나, (b) 렌더 중 그리드가 참조하는 위치 위젯 포인터를
관측해야 한다.

## Q4. 왜 화면 리스트가 비어 보이는가 (itemCount=1인데 행이 안 그려짐)

**답(grade b):** 데이터(행 1개, base 70)는 리스트 모델에 있으나, 그리드 셀 렌더가 유효한
셀 상태/레이아웃을 아직 안 받은 상태일 가능성이 높다. 근거:
- 행 추가 경로 `FUN_00577050`은 std::list 노드를 넣고(`+0x8e4++`) 곧바로 컬럼 루프로
  `FUN_005030f0(grid, colCount*rowIdx+col, flag)`를 돌려 **그리드 셀 상태 바이트**를 세팅한다.
  즉 행 데이터와 셀 상태는 별도 배열이고, 셀 상태가 세팅돼야 그리드가 그 셀을 그린다.
- 렌더는 `FUN_00576b30`→`FUN_005015f0(0xe, grid, …)`→`FUN_00576d40`로 이어지는데,
  `FUN_005755c0`(가드: `in_ECX[6]==0`이면 false, 그리고 `FUN_0050cf40(*in_ECX)` 성공 필요)와
  `*(in_ECX+0x18)` 자식 존재 가드를 통과해야 그린다. 이 가드들 중 하나가 미충족이면
  itemCount가 있어도 셀이 안 그려진다.

**이것이 행 좌표 문제와 같은 뿌리인가:** 부분적으로 그렇다. 좌표가 안 잡히는 것과 셀이 안
그려지는 것 모두 **그리드 셀 레이어가 활성 상태가 아니다**라는 하나의 원인에서 나올 수 있다.
그리드 셀이 실제로 그려지는지(플래그가 세팅되고 `FUN_005755c0` 가드 통과) 여부는 **라이브에서
그리드 셀 플래그 배열(`grid + 0x7c24 + 0x90d0`, 컬럼0)과 가드 필드(`list+0x18`, `list+0x8f8`)를
관측해야 확정**된다. 정적으로는 "가능한 원인"까지만(grade b).

## Q5. listHead04=5000 / listHead08=25 / cellRect / rowWidth 의 정체

| 관측값 | 정체 | grade |
|---|---|---|
| `listHead04 = 5000` | 리스트 ctor `FUN_004fa350`의 리터럴 `mov [esi+4],0x1388`. 용량/한도류 상수. **좌표 아님.** | a (바이트 확인) |
| `listHead08 = 25` | 동일 ctor `mov [esi+8],0x19`. origin walker의 부모-핸들 슬롯 기본값. **행높이 아님 — 가설 기각.** | a (바이트 확인) |
| `cellRect{0,0,232,374}` | 트레이서가 `readPtr(list+0x20)`(=그리드 모델 포인터)을 셀 위젯으로 오인해 `+0x20/+0x24/+0x2c/+0x30`을 rect로 읽은 값. 그리드 모델에는 그 오프셋에 rect가 없으므로 **의미 없음**(우연히 232/374가 읽힌 것). | b |
| `rowWidth = 232` | 위 cellRect의 width 필드에서 파생 → 마찬가지로 **신뢰 불가**. | b |
| `rowHeight = null` / `rowTop = null` | 정적 소스 없음(엔진 계산). **null 유지가 정답.** | a |

**"25 = 행높이" 가설은 grade a로 기각.** 25는 ctor 리터럴 0x19다.

## 행 N 화면 좌표 계산 최종 경로

**정적으로는 산출 불가(확정).** 정적 메모리에 stored origin/rowHeight가 없기 때문. 라이브에서
쓸 수 있는 확정 경로:

- **경로 A (권장):** `FUN_005015f0`(VA `0x005015f0`)에 **함수 경계 훅**을 걸고 `onLeave`에서
  out-struct(param_3, 호출자 스택의 28바이트 버퍼)의 `param_3[2]/param_3[3]`(좌표),
  `param_3[0]/param_3[1]`을 읽어 엔진이 계산한 셀 rect를 캡처한다. 이때 `param_1==0xe`
  (셀 지오메트리 질의)와 `param_2==grid(list+0x20)`인 호출만 채택. → 행/셀별 실제 화면 rect 확보.
- **경로 B (픽셀 우회):** 리스트 메서드 `FUN_00576d40(index)`(VA `0x00576d40`, ECX=list)를
  직접 호출해 인덱스로 선택 상태를 세팅 → 決定 버튼 트리거. 마우스 좌표 자체가 불필요.
  (상태 변경 호출이므로 오라클/QA 드라이브 용도로만.)

두 경로 모두 **함수 경계 훅/직접 호출**이며, 로그인 keysetup 크래시를 유발하는 함수 중간
인라인 훅이 아니다.

## 확정하지 못한 것과, 확정에 필요한 라이브 관측

1. **호스트 위치 위젯의 정체와 절대원점** (Q3): 라이브에서 `FUN_005015f0` 경계 훅으로 out-struct
   좌표를 캡처하거나, 렌더 프레임에서 그리드가 참조하는 위젯 포인터를 backtrace로 관측.
2. **셀이 안 그려지는 정확한 가드** (Q4): 라이브에서 그리드 셀 플래그(`grid+0x7c24+0x90d0`),
   `list+0x18`(자식 존재), `list+0x8f8`(drawn 플래그), `FUN_005755c0` 가드 필드(`list+0x18의+4`,
   `list+0x0` id 유효성)를 렌더 직전/직후 스냅.
3. **행높이·그리드 원점의 실제 픽셀값**: 경로 A로 관측된 rect에서 역산(행0/행1 y차이=행높이).

## 부록: 그리드 객체(`list+0x20`) 관측 오프셋 (라이브 스냅용, grade a 구조)

- 컬럼 c 블록 base = `grid + 0x7c24 + c*0x3f50` (`FUN_005015f0`/필드접근자 case4에서 확인).
- 블록 내: `+0x90d0` 행별 셀 플래그 바이트(0/1/2), `+0x9330` 행 수(600), `+0x9334` 행별 3-dword
