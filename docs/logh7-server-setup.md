# LOGH VII 로컬 서버 구성

현재 저장소에서 증명 가능한 서버 정보는 정적 분석 결과에 한정된다. 레거시 실행 파일은 자동 실행하지 않는다.

## 정적 발견

```powershell
python tools/logh7_pipeline.py discover-server artifacts/logh7-cd/Logh7_mode2_2048.iso --out .omo/ulw-loop/evidence/server-discovery.json
```

현재 발견된 후보:

- 실행 파일: `G7MTClient.exe`, `Gin7UpdateClient.exe`
- 설정 파일: `update.ini`
- URL: `http://www.gineiden.com`
- 업데이트 클라이언트 키: `SERVER_ADDRESS`, `SERVER_PORT`, `PROXY_ADDRESS`, `PROXY_PORT`, `VERSION`, `PORT`, `ADDR`, `TYPE`, `%sSERVER.INI`
- 정적 기본값 후보: `G7MTClient.exe` 안의 `ginei00`, `47900`, `202.8.80.179`; `Gin7UpdateClient.exe` 안의 `47902`, `202.8.80.179`
- 상태: `static-evidence-only`

이 결과는 `data1.hdr`와 `setup.inx`의 문자열 근거다. 실제 게임 서버 프로토콜, 포트, 인증 방식, 클라이언트 접속 절차는 아직 증명되지 않았다. 이를 확인하려면 InstallShield 페이로드를 안전한 작업 디렉터리에 추출한 뒤, `G7MTClient.exe`, `Gin7UpdateClient.exe`, `update.ini`를 샌드박스에서 추가 분석해야 한다.

## 역할 분리 권장안

클라이언트와 서버 작업은 운영체제를 분리한다.

- Windows PC: LOGH VII 설치, 런처, 클라이언트 실행, DirectX/레지스트리/호환성 설정, 실제 접속 동작 캡처를 담당한다.
- macOS 내장 Linux: 리소스/업데이트 서버 분석, Docker 이미지 빌드, 패킷 캡처, 프로토콜 분석, 자동화된 회귀 테스트를 담당한다.
- AWS Docker: 테스트를 제외한 실제 서버 실행 위치다. Windows 클라이언트가 접속할 호스트/포트는 AWS 컨테이너 기준으로 문서화한다.

이 분리는 레거시 Windows 바이너리 실행 위험을 Windows 샌드박스에 가두고, 서버 쪽 반복 분석은 macOS 내장 Linux에서 수행하되 실제 접속 검증과 장기 실행은 AWS Docker로 고정하기 위한 것이다. 서버 프로토콜이 확인되기 전까지 이 저장소의 Node 서버는 게임 서버가 아니라 매니페스트와 업데이트 후보를 로컬에서 검증하는 보조 리소스 서버다.

## AWS Docker 운영 원칙

서버는 테스트 때만 로컬에서 직접 실행한다. Windows 클라이언트가 실제로 접속할 서버는 AWS의 Docker 컨테이너로 실행한다.

운영 단계에서 필요한 산출물:

- Dockerfile 또는 이미지 빌드 명령
- 이미지 태그
- 컨테이너 실행 명령
- AWS host/port
- 보안 그룹 또는 방화벽 규칙
- 로그 확인 명령
- 재시작 절차
- Windows 클라이언트 접속 캡처

## 로컬 리소스 서버

저장소는 발견 매니페스트를 로컬에서 확인하기 위한 의존성 없는 Node HTTP 서버를 제공한다. 기본 바인딩은 `127.0.0.1`이며 외부 인터페이스에 열지 않는다.

```powershell
npm run server:logh7 -- --host 127.0.0.1 --port 4787 --manifest .omo/work/logh7-build/logh7-ko-installed-manifest.json --resource-root .omo/work/logh7-installed
```

다른 터미널에서 확인한다.

```powershell
npm run server:health -- --host 127.0.0.1 --port 4787
curl.exe -i http://127.0.0.1:4787/manifest
curl.exe -i http://127.0.0.1:4787/resources/../../package.json
```

기대 결과:

- `/health`는 HTTP 200과 `ok: true`를 반환한다.
- `/manifest`는 지정한 JSON 매니페스트를 반환한다.
- `/resources/../../package.json` 같은 경로 탈출 요청은 404를 반환하고 저장소 파일을 노출하지 않는다.

### `update.ini` 스키마

`Gin7UpdateClient.exe`의 정적 문자열 근거에 맞춰 서버 매니페스트에 `server.update`가 있으면 `/update.ini`를 생성한다. 스키마가 없거나 필수 키가 빠지면 기존처럼 404를 반환해 추측 설정을 배포하지 않는다.

```json
{
  "server": {
    "update": {
      "VERSION": 131,
      "BASE_DIR": ".\\",
      "SERVER_ADDRESS": "127.0.0.1",
      "SERVER_PORT": 4787,
      "PORT": 47900
    }
  }
}
```

현재 의미:

- `VERSION`: 설치된 `update.ini`에서 확인된 업데이트 버전이다.
- `BASE_DIR`: 설치된 `update.ini`의 기준 디렉터리 값이다.
- `SERVER_ADDRESS`, `SERVER_PORT`: 로컬 업데이트/리소스 서버가 클라이언트에 알려 줄 주소와 포트다.
- `PORT`: `G7MTClient.exe` 정적 문자열에서 확인된 게임 클라이언트 포트 후보 `47900`이다. 실제 게임 프로토콜은 Windows 실행/패킷 캡처로 추가 확인해야 한다.

## 로컬 gameplay TCP capture 서버

`G7MTClient.exe` 정적 문자열에서 확인된 접속 근거는 아직 완성된 게임 프로토콜이 아니라 `47900`, `202.8.80.179`, `ginei00` 후보까지다. 따라서 현재 서버는 로그인/월드 상태를 꾸며서 응답하지 않고, Windows 클라이언트가 보내는 첫 패킷을 JSONL trace로 저장하는 TCP capture stub으로 동작한다.

설치 매니페스트에는 다음 `server.gameplay` 스키마가 포함된다.

```json
{
  "server": {
    "gameplay": {
      "MODE": "tcp-capture-stub",
      "HOST": "127.0.0.1",
      "PORT": 47900,
      "LEGACY_ADDRESS": "202.8.80.179",
      "CLIENT_LITERAL": "ginei00"
    }
  }
}
```

로컬에서 실행한다.

```powershell
npm run server:gameplay -- --host 127.0.0.1 --port 47900 --manifest .omo/work/logh7-build/logh7-ko-installed-manifest.json --trace .omo/ulw-loop/evidence/gameplay-trace.jsonl
```

현재 확인된 동작:

- `server.gameplay` 필수 키가 없으면 서버는 `server.gameplay schema is required for gameplay TCP capture`를 출력하고 포트를 열지 않는다.
- TCP 클라이언트가 `ginei00`을 보내면 trace에 `connection`, `payload`, `close` 이벤트가 JSONL로 기록된다.
- `payload.hex` 값은 `67696e65693030`이며, 이는 ASCII `ginei00`의 원본 바이트다.
- 아직 프로토콜 응답은 만들지 않는다. 실제 Windows 클라이언트 패킷을 모은 뒤 login/session/world 스키마를 확정한다.

캡처된 JSONL은 frame header, 방향, message code 단위로 정규화한다.

```powershell
python tools/logh7_pipeline.py gameplay-trace-analyze .omo/ulw-loop/evidence/gameplay-trace.jsonl --out .omo/ulw-loop/evidence/gameplay-packets.json
```

분석 스키마:

- `summary.connections`, `summary.payloadPackets`, `summary.responsePackets`, `summary.closes`: trace 이벤트 집계다.
- `packets[].direction`: `payload`는 `client-to-server`, `response`는 `server-to-client`로 기록한다.
- `packets[].frame.declaredPayloadLength`: frame 첫 2바이트 big-endian payload length다.
- `packets[].frame.messageCodeHex`: frame 3-4번째 바이트의 big-endian message code다.
- `packets[].frame.kind`: 현재 `0x0034` 28-byte frame은 `observed-login-request`, `0x0035` response candidate는 `phase3-response-candidate`, phase1-derived `0x0035` 이후 실제 클라이언트가 보낸 `0x0036` 12-byte frame은 `observed-post-phase3-client-packet`, `0x0030` 52-byte frame은 `observed-post-handshake-client-packet`으로 분류한다.

실제 Windows 로그인 UI에서 관찰된 요청은 `001a0034...` 형태의 28-byte `0x0034` frame이다. configured response manifest로 서버가 돌려준 정적 후보는 `00120035...` 형태의 20-byte `0x0035` frame으로 분석된다. live `0x0034` request에서 파생한 GUID transport-key 기반 `0x0035` 응답은 real client를 이전 `exchange_key_phase3 encipher` 실패 지점 너머로 진행시켰고, 같은 연결에서 `000a0036...` 및 `00320030...` post-handshake client packet을 관찰했다. 아직 이 두 패킷에 대한 서버 응답은 record-only 상태다.

## 클라이언트 프로토콜 카탈로그

`G7MTClient.exe`에는 네트워크 기본값뿐 아니라 login/session/world 메시지 심볼과 robot 실행 모드 문자열이 들어 있다. 다음 명령은 실행 파일의 ASCII 문자열만 근거로 서버 스키마 후보를 JSON으로 만든다.

```powershell
python tools/logh7_pipeline.py client-protocol-index .omo/work/logh7-installed/exe/G7MTClient.exe --out .omo/ulw-loop/evidence/g004-client-protocol-index.json
```

현재 확인된 정적 스키마:

- `defaults.account`: `ginei00`
- `defaults.loginServerPort`: `47900`
- `defaults.loginServerAddress`: `202.8.80.179`
- `commandLineModes[0].usage`: `usage : >robot <login-server address> <login-server port> <session-server name>`
- login 메시지 예: `LobbyLoginRequest`, `LobbyLoginOK`, `LobbySessionLoginRequest`
- `messageCodes.login`에는 바이너리 등록 테이블 순서상 `LobbyLoginRequest=0x2000`, `LobbyLoginOK=0x2001`, `LobbyLoginNG=0x2002`가 들어간다.
- `cipherHandshake.phases`에는 `[mpsCipherManager] exchange_key_phase*` 진단 문자열이 들어간다.
- `cipherHandshake.transportCodes`에는 현재 관측/역추적된 cipher transport 코드 `0x0034` client-to-server request, `0x0035` server-to-client `exchange_key_phase3` response가 들어간다.
- `cipherHandshake.transportDispatch`에는 jump table 근거의 내부 큐 코드가 들어간다. 현재 `0x0030 -> 0x0300/0x0301`, `0x0034 -> 0x0405`, `0x0035 -> 0x0406`, `0x0036 -> 0x040c`이며 모두 client offset `0x35837e`의 cipher-enabled flag를 gate로 탄다. `0x0030` handler는 추가로 timestamp/gettick 결과를 `client+0x357eac`에 저장한다.
- `cipherHandshake.phase3Param2Observations`에는 실제 Windows 클라이언트 probe에서 얻은 `0x0035` body length별 phase3 진단 분기가 들어간다.
- `cipherHandshake.adjacentTransportProbe`에는 `0x0034`, `0x0036` 주변 코드 probe가 같은 조건에서 `mpsCipherManager` stderr 없이 EOF로 끝났다는 관찰이 들어간다.
- `cipherHandshake.requestDerivedPhase3Probe`에는 live `0x0034` request body에서 파생한 `0x0035` 응답 후보별 phase3 진단이 들어간다. request body echo 계열은 `broken data`까지 도달했지만 로그인 이후 진행은 없었다.
- `cipherHandshake.phase3DecodedPayloadLayout`에는 phase3 decode 이후 payload의 정적 역추적 구조가 들어간다. `0x645660` phase3 함수는 network-order checksum, encipher key length/data, decipher key length/data, sequence dword 순서로 decoded payload를 검사한다.
- `cipherHandshake.phase1DecodedPayloadLayout`에는 phase1 outbound decoded payload 구조가 들어간다. `0x645180` phase1 함수는 checksum, key length/data, sequence dword 순서로 payload를 만든 뒤 manager `+0x04` child codec encoder로 raw transport body를 만든다.
- `cipherHandshake.phase2DecodedPayloadLayout`에는 phase2 inbound decoded payload 구조가 들어간다. `0x6452f0` phase2 함수는 checksum, remote key length/data, stored key length/data, sequence dword 순서로 payload를 읽고, remote key를 manager `+0x0c` child codec에 적용한다.
- `cipherHandshake.decodedPayloadImplementation`에는 현재 서버 툴링에서 구현된 decoded payload builder/parser 범위가 들어간다. phase1/phase2/phase3 decoded payload, checksum helper, child-codec encrypted transport frame wrapping은 테스트된 상태지만, 실제 live phase key material을 서버 세션 상태에 연결하는 작업은 남아 있다.
- `cipherHandshake.phase3ChecksumAlgorithm`에는 첫 2바이트 checksum을 제외한 decoded payload 구간을 little-endian dword XOR과 trailing byte XOR으로 줄인 뒤 high16 xor low16으로 fold하는 비교 규칙이 들어간다.
- `cipherHandshake.phase3WireEncoding`은 아직 `unresolved`다. phase3 함수는 이 layout을 적용하기 전에 cipher vtable decode path를 먼저 호출하므로, raw `0x0035` TCP body와 decoded payload는 같은 구조로 취급하면 안 된다.
- `cipherHandshake.phase3DecodedPayloadRawWireProbe`에는 checksum-correct decoded payload를 raw `0x0035` body로 보낸 실제 Windows 클라이언트 probe 결과가 들어간다. 이 후보들은 로그인 이후로 진행하지 않았고 raw wire encoding 미해결 판단을 지지한다.
- `cipherHandshake.phase3ProcessingPipeline`에는 phase3 함수의 vtable 호출 순서가 들어간다. raw param2 decode는 manager offset `+0x04` vtable slot `0x10`에서 resolved target `0x614460`, encipher key 비교는 `+0x08` slot `0x08`, decipher key 적용은 `+0x0c` slot `0x04`, phase3 reply encode는 `+0x04` slot `0x0c`에서 resolved target `0x614100`을 탄다.
- `cipherHandshake.phase3ChildCodecVtable`에는 phase3 manager 생성자 `0x6450e0`에 주입되는 child codec 생성자 `0x613fc0`, vtable `0x681fc8`, slot `0x0c -> 0x614100`, slot `0x10 -> 0x614460`이 들어간다. `0x614460`은 raw body 길이가 8-byte aligned가 아니면 실패하고 block transform 경로로 진입한다.
- `cipherHandshake.phase3ChildCodecBlockTransform`에는 child codec의 block transform 구조가 들어간다. 현재 정적 근거상 Blowfish-like Feistel 구조이며 8-byte block, 16 rounds, 18 dword P-array, 4 x 256 dword S-box, static table xor mask `0x91`, encode zero padding, decode strict 8-byte alignment를 사용한다.
- `cipherHandshake.childCodecStaticTableImplementation`에는 현재 서버 툴링이 실제 `G7MTClient.exe` PE에서 추출할 수 있는 child codec static table과 block codec 구현 범위가 들어간다. P-array VA `0x007b6ae4`, S-box VA `0x007b6ba8`, xor mask `0x91`, round function, key schedule, block encode/decode helper, phase3 encrypted frame builder, live phase1 request replay는 구현됐고, phase2/phase3 response key material을 live response state에 연결하는 작업은 남아 있다.
- `cipherHandshake.phase3ConfiguredResponseSchema`에는 `server.gameplay.loginResponse` manifest schema가 들어간다. gameplay 서버는 이 schema가 명시된 경우에만 관측된 `0x0034` frame에 대해 전체 transport frame hex를 응답으로 쓰며, 기본 manifest에서는 여전히 fabricated login/session response를 내지 않는다.
- `cipherHandshake.phase3ChildCodecKeyFlow`에는 child codec key schedule 진입점과 key 저장 형식이 들어간다. child slot `0x04` wrapper `0x6140c0`은 `0x613ad0` key schedule에 raw key bytes/length를 넘기고, 성공 후 `0x614810`에서 key bytes를 `xor 0x17` 저장 이미지로 보관한다. 저장된 key는 `codec+0x04` data pointer와 `codec+0x08` length word로 읽힌다.
- `cipherHandshake.requestBodyKeyAnalysis`에는 observed `0x0034` request body를 child codec으로 디코드해 phase1/phase2/phase3 decoded payload로 분류하는 분석 도구와 결과가 들어간다. 현재 `abcd`, `ginei00`, `dummy`, `127.0.0.1`, `47900` 정적 후보 키뿐 아니라 `keySetupWrapper` login GUID/session raw key와 `keyReadHelper` phase1 outbound stored/raw key도 decoded payload checksum을 통과하지 못했다. `runtime-child-encode-log-patch`는 `0x614100` entry에서 phase1 plaintext와 active transport codec key image를 잡았고, `runtime-child-encode-post-log-patch`는 `0x6452cc` post-call에서 실제 `0x0034` frame body와 같은 encoded output을 캡처했다. corrected Python replay 이후 다음 trace point는 post-handshake `0x0036`/`0x0030` client packet 분류와 응답 구현이다.
- `cipherHandshake.phase1DerivedResponseProbe`에는 live `0x0034` request에서 phase1 key/sequence를 파싱하고 GUID transport key로 `0x0035` response를 만든 실제 Windows probe 결과가 들어간다. 이 응답은 stderr 없이 클라이언트를 phase3 encipher failure 이후로 진행시켰고 후속 `0x0036`, `0x0030` frame을 관찰하게 했다.
- `cipherHandshake.postHandshakeDispatchFindings`에는 `transport-dispatch-index`가 실제 `G7MTClient.exe` jump table `0x004b864c`와 dispatch tail `0x004b78ef`에서 복원한 후속 transport mapping이 들어간다. observed `0x0036`은 internal `0x040c`, observed `0x0030`은 internal `0x0300/0x0301` pair로 이어진다.
- `cipherHandshake.postHandshakeHandlerFindings`에는 `post-handshake-handler-index`가 실제 `G7MTClient.exe`에서 복원한 후속 handler 단서가 들어간다. internal `0x0300`은 같은 internal dispatch routine 안에서 직접 payload handler로 들어가지 않는 paired request-side route이고, internal `0x0301` ack/timing handler가 `0x0030` body의 첫 dword를 읽고 `client+0x432418`, `client+0x357eb0`, `client+0x357ea8`에 기록한다. internal `0x040c` phase4 builder는 client offsets `0x30a`, `0x30b`, `0x312`, `0x313`, `0x314`, `0x30c`, `0x30d`, `0x30e`, `0x310`, `0xd14`를 `0x0036` payload로 serialize하고, `0x004c1949` send trigger가 internal `0x040c`를 큐잉한다.
- `cipherHandshake.postHandshakeFollowupAckProbe`에는 실제 Windows 클라이언트가 `0x0036`을 보낸 직후 서버가 raw `0x0030` dword ack candidate `0006003000000000`을 보낸 probe 결과가 들어간다. 이 후보는 `mpsCipherManager` stderr나 즉시 crash를 만들지는 않았지만, 클라이언트는 여전히 자체 `0x0030` 52-byte frame을 보낸 뒤 EOF로 끝났다. 따라서 이 frame은 기본 서버 응답으로 승격하지 않고, 다음 작업은 client `0x0030` body decode와 internal `0x0300/0x0301` response semantics 확정이다.

다음 명령은 debugger attach 없이 file-backed 계측 패치를 준비하기 위한 guard schema를 만든다. 각 target은 VA, PE file offset, 원본 16-byte signature, 역할을 포함하며 signature가 어긋나면 실패한다. `probePlan`에는 logging trampoline을 둘 executable code cave와 파일 출력에 필요한 imported function IAT slot도 포함된다.

- `keySetupWrapper`: VA `0x006140c0`, file offset `0x002140c0`; raw key bytes/length가 child codec key schedule로 들어가는 wrapper다.
- `keyStoreHelper`: VA `0x00614810`, file offset `0x00214810`; raw key를 `xor 0x17` stored image로 저장한다.
- `keyReadHelper`: VA `0x006148a0`, file offset `0x002148a0`; stored key image를 읽고 raw key로 복원한다.
- `childCodecEncode`: VA `0x00614100`, file offset `0x00214100`; phase1/phase3 decoded payload를 child codec transport body로 encode하는 vtable slot `0x0c` target이다.
- `phase1ChildEncodePostCall`: VA `0x006452cc`, file offset `0x002452cc`; phase1 caller가 child codec encode에서 돌아온 직후, output holder를 해제하기 전 encoded body를 캡처하는 callsite다.

다음 명령은 같은 KLG2 record format을 사용하되 `keySetupWrapper` entry를 hook한다. 이 record의 `returnAddress`는 key setup wrapper를 호출한 간접 callsite 직후 주소이므로, `keyStoreHelper` return address만으로 구분되지 않던 manager field 역할을 분리하는 데 쓴다.


현재 확인된 runtime key log record 스키마:

- `magic`: ASCII `KLG2`
- `event`: `1` for `keyStoreHelper` entry
- `eventName`: `keyStoreHelperEntry` 또는 `keySetupWrapperEntry`
- `returnAddress`: helper caller return address
- `helperReturn`: known return-address label, currently `keySetupWrapper.storeKeyReturn` for `0x006140ef`, `loginGuidKeySetup.returnAfterKeySetup` for `0x0061285c`, `loginSessionKeySetup.returnAfterKeySetup` for `0x00612d0b`, `phase2InboundApply.returnAfterKeySetup` for `0x00645483`, and `phase3Apply.returnAfterKeySetup` for `0x00645944`
- `codec`: child codec instance pointer
- `keyPointer`: raw key pointer observed at helper entry
- `keyLength`: raw key byte length
- `copyLength`: copied key byte count, capped at 64
- `keyBytes64`: copied raw key bytes

실제 Windows 클라이언트 probe는 로그인 UI에서 `ginei00`/`dummy` 입력 후 28-byte `0x0034` request를 보냈고, 같은 실행에서 `KLG2`/`CLG2` records를 기록했다. `keyStoreHelper` hook에서는 두 GUID key와 하나의 16-byte per-run key가 모두 `returnAddress=0x006140ef`, `helperReturn=keySetupWrapper.storeKeyReturn`으로 수렴해 manager 역할을 구분하지 못했다. `keySetupWrapper` entry hook은 no-op jump만 주입하면 안전했지만, record buffer를 executable code cave에 쓰는 순간 현재 Windows에서 access violation이 발생했다. live `VirtualProtectEx`로 cave page를 `PAGE_EXECUTE_READWRITE`로 바꾸면 같은 hook이 살아서 `keySetupWrapperEntry` metadata를 기록했고, 최종 파일 패치본은 PE section `MEM_WRITE` 보정으로 같은 효과를 영구화한다. prepatched `G7MTClient.kwrap.copy.exe` 표면 QA는 login request `0x0034`와 276-byte `logh7_keylog.bin` 생성을 확인했다. `runtime-keylog-read`는 두 38-byte GUID setup record를 `returnAddress=0x0061285c`, `helperReturn=loginGuidKeySetup.returnAfterKeySetup`, 16-byte session key setup record를 `returnAddress=0x00612d0b`, `helperReturn=loginSessionKeySetup.returnAfterKeySetup`으로 파싱하고 raw key bytes까지 복사한다. 추가 prepatched `G7MTClient.kread.fieldfix.exe` probe는 `keyReadHelperEntry` 1건을 기록했으며 `returnAddress=0x006451a2`, `helperReturn=phase1OutboundRead.returnAfterKeyRead`, stored image `315ab37d3a44955490c29ba1abdb3f89`, raw xor-0x17 key `264da46a2d53824387d58cb6bccc289e`를 확인했다. prepatched `G7MTClient.childencode.keycapture.exe` probe는 `childCodecEncodeEntry` 1건을 기록했고 `0x0034` request 직전 phase1 plaintext, generated phase1 key, active transport codec GUID key image를 확인했다. prepatched `G7MTClient.childpost.exe` probe는 `childCodecEncodePostCall` 1건을 기록했고 post-call output `88396949581bcc872316b86f23a92d45014cbc56d722012b`가 같은 실행의 `0x0034` request body와 일치함을 확인했다.
- session 메시지 예: `SSLoginRequest`, `SSLoginOK`, `SSGameLoginRequest`
- world 메시지 예: `RequestWorldInitialize`, `ResponseWorldInitialize`, `RequestGridInitialize`

이 카탈로그는 서버가 어떤 메시지 계층을 준비해야 하는지 알려 주지만, wire format을 증명하지는 않는다. 따라서 아직 `LobbyLoginOK` 같은 응답을 생성하면 안 된다. 실제 Windows 클라이언트가 보낸 바이트를 capture trace로 확보한 뒤에만 login/session 응답 스키마를 확정한다.

현재 실행 관찰:

- `G7Start.exe`는 게임 접속 런처가 아니라 CD auto-starter/설치 런처다. 버튼은 설치, 제거, PDF 매뉴얼, 종료이며 `SETUP.EXE`와 `\DirectX9` 경로 문자열을 가진다.
- `G7Start.exe`는 `DSETUP.dll` import가 있고, 설치 트리에 이 런타임이 없으면 시스템 오류 대화상자에 막힌다. `build-installed`는 ISO 루트의 `dsetup.dll`, `dsetup32.dll`을 설치 트리 루트의 `DSETUP.dll`, `DSETUP32.dll`로 포함한다.
- `G7MTClient.exe`를 `exe` 디렉터리에서 직접 실행하면 실제 게임 로그인 화면까지 도달한다. 현재 로그인 화면의 일본어 UI는 깨져 보이고, 실행 중 `exe/String.txt`가 0바이트로 바뀌므로 실행 QA 뒤에는 `build-installed`로 설치 트리를 다시 생성한다.
- `build-installed`는 현재 Windows 실행 보조 파일 `setup-local.ps1`, `launch-client.ps1`, `WINDOWS-COMPATIBILITY.txt`를 설치 트리 루트에 생성한다. `setup-local.ps1`은 per-user registry/AppCompatFlags를 만들고, `launch-client.ps1`은 `exe/String.txt`를 백업본에서 복원한 뒤 `exe` working directory로 클라이언트를 실행한다. 최종 `exe/G7MTClient.exe`는 playable manifest를 검증한 EXE여야 하며, `LOGH_STRAT_GRID_EARLY=1` 기본 실행 때문에 `menufix`, `dlgfix`, `earlygrid-ringclear` 스택이 필요하다.
- gameplay 서버 readiness를 TCP 연결로 확인하면 trace에 readiness probe가 기록된다. Windows 클라이언트 handshake를 증명할 때는 서버 stdout의 `listening` 라인을 기다리고 trace를 readiness probe로 오염시키지 않는다.
- 과거 `build-installed`는 `G7MTClient.exe` 안의 단일 `202.8.80.179` 로그인 서버 literal을 생성된 설치 트리에서만 `127.0.0.1`로 NUL padding 치환했다. 현재는 그 치환 결과를 포함한 canonical playable EXE를 최종 클라이언트로 배포하고, manifest에 SHA와 patch stack을 남긴다. 이 redirect 뒤 실제 로그인 UI에서 `ginei00`/`dummy`를 입력하고 로그인 버튼을 누르면 `127.0.0.1:47900`으로 첫 28바이트 payload가 들어왔다.
- 관찰된 첫 payload의 frame header는 big-endian `declaredPayloadLength=26`, `messageCode=52`이고 전체 길이는 28바이트다. 서버는 이를 `observed-login-request`로 trace에 기록하지만, 아직 서버 응답 바이트가 관찰되지 않았으므로 login/session 성공 응답은 보내지 않는다.

- G037 real-client body/encoding probe는 session bootstrap builder와 world-init probe server에 `bootstrap-encoding` 및 `bootstrap-body-hex` 변형을 추가했다. 기본값은 기존 `phase1-child-codec`/`01`이며, 실험용으로 raw frame body를 보낼 수 있다. 실제 Windows client에서 `after-0030`, raw `01`은 frame `0003000101`/`0003000301`을 보냈고, raw `01000000`은 frame `0006000101000000`/`0006000301000000`을 보냈다. 두 변형 모두 trace summary는 `sessionBootstrapResponseCandidates=2`, `postSessionBootstrapClientPackets=0`이고 live scan은 `ssLoginOkFlag=0`, `ssGameLoginOkFlag=0`, `cipherGate=0`, `responseWorldInitialized=0`, `responseGridInitialized=0`, `activeRecords=[]`였다. 따라서 실패 원인은 단순 phase1-child-codec wrapping이나 1-byte body 길이만이 아니며, 다음 분석은 low transport dispatch 전 단계의 session-state/framing prerequisite 또는 다른 response family를 찾아야 한다.
- G038 `session-bootstrap-index`는 `0x004b852b`의 low transport queue append 경로를 추가로 복원한다. 실제 PE evidence `.omo/ulw-loop/evidence/g038-session-bootstrap-queue-schema.json`에 따르면 transport handler tail은 runtime manager global `0x007c25f4`가 non-null일 때 `client+0x357ec0` pending count를 읽고, 12-byte entry stride로 `client+0x357ec4`에 queued internal code, `client+0x357ec8`에 paired internal code, `client+0x357ecc`에 payload/context pointer를 저장한다. 첫 entry는 runtime manager vtable `+0x18` 호출로 알리고, queue capacity는 100개다. 따라서 다음 trace point는 같은 TCP connection에 frame을 쓰는 수준이 아니라, 실제 클라이언트 runtime에서 이 manager-backed queue append가 일어나는 조건을 계측하거나 그 조건에 맞는 response path를 찾아내는 것이다.
- G039 live scan은 G038 queue schema를 실제 process memory scan에 연결했다. `tools/logh7_live_entity_scan.py`는 이제 `client+0x357ec0`의 `transportQueueCount`와 최대 8개 pending entry를 `transportQueueEntries[]`로 읽는다. 실제 Windows client probe `.omo/ulw-loop/evidence/g039-real-client-queue-scan-probe.json`은 `after-0030`, raw `01000000` bootstrap 후보와 world/grid 후보를 보낸 뒤 `transportQueueCount=0`, `transportQueueEntries=[]`, `ssLoginOkFlag=0`, `ssGameLoginOkFlag=0`, `cipherGate=0`, `responseWorldInitialized=0`, `responseGridInitialized=0`, `activeRecords=[]`를 기록했다. 이 scan은 queue가 순간적으로 append 후 drain됐을 가능성까지 배제하지는 않지만, 후보 송신 뒤 관측 가능한 pending low-response queue가 남지 않았고 session/world flags도 변하지 않았다는 runtime evidence다.
- G040 `runtime-patch-targets`는 `lowTransportQueueAppend`를 file-backed instrumentation guard target으로 포함한다. 실제 PE evidence `.omo/ulw-loop/evidence/g040-runtime-patch-targets-queue.json`은 VA `0x004b852b`, file offset `0x000b852b`, 원본 16-byte signature `8a450884c00f84db00000083fbff7517`, role `low transport pending response queue append path for queued/paired internal codes`를 기록한다. 이는 G039의 scan 한계를 보완할 다음 계측 지점이며, 아직 실제 hook patch와 runtime record는 만들지 않았다.
- G041 queue append instrumentation은 분기 재배치 위험이 있는 `0x004b852b` 대신 store block `0x004b8552`를 `lowTransportQueueAppendStore` target으로 추가했다. 실제 PE signature는 `8b87c07e35008d14408b4510898497cc`이고, `runtime-queue-append-log-patch`는 `logh7_queue_append.bin`에 32-byte `QLG1` record를 append하도록 패치한다. record layout은 `clientPointer`, `queuedInternalCode`, `pairedInternalCode`, `payloadOrContextPointer`, `queueCount`, `transportFlag`다. 패치된 실제 Windows client QA는 기존 후보 sequence(`0x0034`, server `0x0035`, client `0x0036`, client `0x0030`, server raw `0x0001/0x0003`, world/grid `0x0013/0x0014`)를 재현했지만 `logh7_queue_append.bin`이 생성되지 않았다. 따라서 현재 후보 패킷은 socket trace에는 보이지만 클라이언트의 low transport queue append path까지 도달하지 못한다. 다음 분석은 payload/body 변형이 아니라 수신 frame parser, manager prerequisite, 또는 append path 직전 branch 조건이다.
- G042 queue append entry instrumentation은 `0x004b852b`의 원래 `mov al,[ebp+8]; test al,al; je 0x004b8611` 조건분기를 trampoline에서 직접 보존하면서 entry record를 남긴다. `runtime-queue-entry-log-patch`는 `logh7_queue_entry.bin`에 32-byte `QEG1` record를 append하고, record layout은 `clientPointer`, `queuedInternalCode`, `pairedInternalCode`, `payloadOrContextPointer`, `queueCount`, `transportFlag`, `branchPolicy:transportFlag0SkipsTo0x004b8611`이다. 패치된 실제 Windows client QA는 G041과 같은 후보 sequence를 재현했지만 `logh7_queue_entry.bin`도 생성되지 않았다. 그러므로 실패 지점은 append 내부의 `transportFlag == 0` branch가 아니라, 현재 서버 후보 frame이 low transport append routine 자체를 호출하지 못하는 더 앞단이다. 다음 trace point는 socket receive frame parser, decoded low-response family discriminator, 또는 runtime manager object prerequisite이다.
- G043 dispatch-tail prerequisite 분석은 `0x004b78ef`가 `mov eax,[0x007c25f4]; test eax,eax; jne 0x004b852b`로 runtime manager global을 검사함을 확인했다. `tools/logh7_live_entity_scan.py`는 이제 `runtimeManagerPointerHex`를 live scan에 포함한다. 실제 Windows client QA에서 같은 후보 sequence 뒤 `runtimeManagerPointerHex=0x00000000`, `transportQueueCount=0`, session/world/grid flags all `0`이었다. 따라서 G042의 append entry 미호출은 receive frame이 무조건 다른 parser로 갔다는 결론보다 더 구체적으로, 현재 진행 상태에서 runtime manager object가 아직 초기화되지 않아 dispatch tail이 append로 점프할 수 없는 상태임을 의미한다. 다음 스키마 목표는 `0x007c25f4`를 non-null로 만드는 선행 초기화 path 또는 그 manager 생성 response family다.
- G044 `runtime-manager-index`는 `0x007c25f4` lifecycle을 실제 PE에서 추출한다. 생성 path는 `0x004ad94f`에서 `runtimeManagerGlobal = ebp`를 저장하고, 직전 `0x00612570` allocation call은 size `0x7530`을 사용한다. 생성 직후 callback 후보는 `0x004adeb0`, `0x004add60`, `0x004adf60`이고, `0x004adb09` destructor path는 global을 `0`으로 clear한다. 따라서 현재 접속 path에서 manager가 0인 이유는 단순 서버 response code 누락만이 아니라, 클라이언트가 이 manager constructor path를 아직 실행하지 않은 상태다. 다음 trace point는 `0x004ad94f` constructor 호출을 유발하는 packet/UI path를 찾는 것이다.
- G045 constructor-store hook은 G044의 결론을 정정한다. `runtime-manager-log-patch`는 `0x004ad94f` store를 file-backed hook으로 감싸 `logh7_runtime_manager.bin`에 32-byte `RMG1` record를 남긴다. 패치된 실제 Windows client QA에서 같은 로그인/후보 sequence 중 `RMG1` record 1개가 생성됐고 `managerPointer=0x05402a10`, `globalBefore=0x00000000`을 기록했다. 하지만 probe 후 live scan은 다시 `runtimeManagerPointerHex=0x00000000`, `transportQueueCount=0`이었다. 즉 현재 경로는 manager constructor를 호출하지만, low transport response 후보가 처리될 때까지 manager global이 유지되지 않는다. 다음 trace point는 constructor caller가 아니라 `0x004adb09` destructor/global-clear hook 또는 manager lifetime을 끝내는 close/error path다.
- G046 destructor-clear hook은 G045의 lifetime 가설을 실제 실행에서 확인한다. `runtime-manager-clear-log-patch`는 `0x004adb09` clear를 file-backed hook으로 감싸 `logh7_runtime_manager_clear.bin`에 32-byte `RMC1` record를 남긴다. 패치된 실제 Windows client QA에서 같은 로그인/후보 sequence 중 `RMC1` record 1개가 생성됐고, clear 직전 `globalBefore=0x05432a10`이었다. 같은 실행의 packet trace는 client `0x0034`, server `0x0035`, client `0x0036`, client `0x0030`, server raw `0x0001/0x0003`, server `0x0013/0x0014`, close 순서를 기록했지만 probe 후 live scan은 `runtimeManagerPointerHex=0x00000000`, `transportQueueCount=0`이었다. 따라서 패킷은 실제로 오가고 있으나 현재 서버 후보가 manager-backed low transport queue로 들어가기 전에 manager lifetime이 끝난다. 다음 trace point는 clear caller/timing을 store hook과 같은 실행에서 상관시키거나, close/error path를 끝내는 누락 response prerequisite을 찾는 것이다.
- G047 `runtime-manager-index`는 clear caller evidence를 추가한다. 실제 PE에서 manager primary vtable `0x0066e0fc`의 slot0은 `0x004adaa0` virtual destructor wrapper이고, wrapper의 direct body call은 `0x004adaa3 -> 0x004adac0`이다. clear body는 `0x004adaf6`에서 `edi+0x40` member를 로드하고 `0x004adb01 -> 0x00403c50` shutdown call을 실행한 직후 `0x004adb09`에서 global을 0으로 clear한다. 따라서 G046의 clear는 임의의 response handler direct path가 아니라 manager 객체 virtual destructor lifecycle이며, 다음 trace point는 virtual destructor entry와 socket close/error path를 같은 실행에서 상관시키는 것이다.

## MsgDat 스키마 인덱스

설치 트리의 `data/MsgDat/*.dat` 파일은 평문이 아니라 `HFWR` 또는 `GFWR` magic을 가진 바이너리 컨테이너다. 서버와 한글화 데이터 스키마는 이 컨테이너를 먼저 인덱싱한 뒤 별도 패치 포맷으로 확장한다.

```powershell
python tools/logh7_pipeline.py msgdat-index .omo/work/logh7-installed/data/MsgDat --out .omo/ulw-loop/evidence/msgdat-index.json
```

현재 인덱스 스키마:

- `files[].path`, `files[].size`, `files[].magic`: 원본 파일과 컨테이너 종류다.
- `files[].layout`: 클라이언트 로더 기준 레이아웃이다. `HFWR`는 `textPointerCount`/`offsetTableCount`/`payloadOffset`을 갖고, `GFWR`는 `recordCount`/`payloadOffset`을 갖는다.
- `files[].records[]`: 실제 클라이언트가 만드는 레코드 ID 단위 문자열이다. `HFWR`는 NUL 종료 CP932 문자열, `GFWR`는 길이 prefix가 붙은 UTF-16LE 문자열이다.
- `files[].tokens[]`: `$xcommand$` 같은 ASCII 필드 토큰과 원본 byte offset이다.
- `files[].textCandidates[]`: 보조용 CP932 후보 문자열과 원본 byte offset이다. 실제 write-back과 서버 message ID 매핑은 `records[]`를 우선한다.

로그인/로비 에러 응답 implication:

- `LobbyLoginNG`/`LobbySessionLoginNG` wire body는 문자열 자체가 아니라 `error_code u8`, `message_count u8`, `u16 message_id[]` 형태다.
- `constmsg.dat`의 로그인 관련 message ID: `0x0bff`=`ログインに失敗しました。`, `0x0c00`=`バージョンが違います。`, `0x0c02`=`あなたはすでにログインしています。`, `0x0c03`=`そのアカウントは存在しません。`, `0x0c04`=`パスワードが一致しません。`, `0x0c07..0x0c0b`=`認証に失敗しました。`.
- 서버는 인증 실패 사유를 직접 일본어 문자열로 보내지 않고, 위 message ID 배열로 매핑해야 한다. 한국어/영어 현지화도 같은 ID를 유지하고 MsgDat overlay만 바꾸는 쪽이 안전하다.
