# LOGH VII 콘텐츠 검증 — 적대적 재검증(Adversarial Re-verification)

> **대상**: `docs/logh7-content-verify.md` §1(동맹 함선스탯 교정 #1~#12) + §3(initial-deployment 5포인트)
> **방법**: content-verify의 주장값을 **신뢰하지 않고**, `gin7manual.pdf`를 PyMuPDF(fitz 1.27.2)로 **좌표 독립 재추출**.
>  - rot=90 세로조판(tategaki) 페이지는 `get_text('words')` 셀 x/y 빈닝 + `get_text('rawdict')` 라벨 문자 방향(dir 벡터)으로 행순서 독립 결정, `fitz.Matrix(4,4)` 4배 렌더 PNG 육안 교차확인.
>  - 표준컬럼 규칙은 known-canonical 제국함(A76 공작함, A74 수송함, A78 양륙함, SS75 전함)으로 캘리브레이션.
>  - p75 초기배치는 rotation=0 정상 가로 텍스트(과제의 '90도 회전' 전제 부적용)라 좌표 비닝으로 완전 재현.
> **원칙**: confirm(high/medium)만 반영 권고. refute/uncertain은 라이브/추가 RE 대기. **단일추출 신뢰 금지였던 항목이 독립검증을 통과했는지** 항목별로 명시.
> **총계**: **confirm 14 / refute 2 / uncertain 1** (검증 항목 17건 = 함선 12 + 초기배치 5).

## (b) 적용 가능 — confirm(high/medium) 목록

> **단일추출 신뢰 금지였던 항목의 독립검증 통과 여부**: content-verify §1의 동맹 11엔트리는 OCR linear-flatten으로 confidence=low/none이었던 항목이다. 아래 confirm 항목은 **두 번째(이번) 적대적 좌표 재추출이 첫 추출과 독립적으로 동일 결론에 도달**했으므로, 단일추출 신뢰가 아니라 **2회 독립추출 합치**로 격상 근거가 성립한다. (변형열 차용 필드는 별도 단서 — 아래 ⚠️ 참조.)

### C-1. refute — 적용하지 말 것 (현행 json이 이미 정확)

| 항목 | content-verify 주장 | 적대검증 반박 근거 | 처리 |
|------|--------------------|-------------------|------|
| **#8 공작함 전체** | 장갑 24/14/8→22/12/8, 速度 18000→20000 | 22/12/8·20000은 工作艦**II**('경량구조·기동성↑') **변형값**. I(標準)=24/14/8·18000이며, 제국 A76(標準 26/16/10·18000) 캘리브레이션상 標準은 I/병합행에 위치. 주장은 **II 변형을 標準으로 오치환**. **현행 json(24/14/8·18000)이 이미 標準으로 정확** | **현행 유지**. ship-stats.json 변경 금지 |
| **#11 양륙함 速度** | 15000→18000 | 18000은 揚陸艦**II**('신형기관·고속화')·IV 변형값. I(795년형 標準)=15000. 제국 A78(標準 speed 12000) 동일규칙. **현행 15000이 標準으로 정확** | **速度는 현행 유지**. (단 같은 항목의 物資+30은 confirm → B-1에서 별도 반영) |

> 교훈(변형열 함정): content-verify §1은 일부 항목에서 **base/標準(Ⅰ)행이 아니라 軽量/高速化 변형(Ⅱ·Ⅳ)열의 값을 표준으로 잘못 취했다**. #8·#11 速度가 그 사례. 제국 동급함(A76/A78) 표준값 위치로 캘리브레이션하면 변형열이 가려진다. **#3·#6의 변형열 차용 필드(ビーム/対空/修理)도 같은 위험군**이라 med 격상에 그친다.

### C-2. uncertain — 라이브/추가 RE 대기

| 항목 | 주장 | 불확실 사유 | 대기 조건 |
|------|------|------------|----------|
| **#3 전함 ビーム 64** | 90→64 | 64는 base행이 아닌 **戦艦IV 변형열 차용**(base 공란). 방향은 타당(제국 SS75 ビーム=64와 매그니튜드 정합)하나 표준값으로의 직접 출처가 없음 | 동맹 전함 ビーム base의 실제 캐논값을 클라 함선스탯 소비처(0x030b ship-class / 0x0323) 라이브 RE로 확정 후 적용. 그때까지 90 오염값은 제거하되 64는 잠정(med) |

## (d) 초기배치 신규파일 생성 권고

**권고: `content/initial-deployment.json` 신규 생성 (가능).**

- 5포인트 전건 **confirm-high**(refute 0, uncertain 0). 과제의 '90도 회전 세로조판' 전제는 p75엔 부적용(rotation=0 정상 가로 텍스트)이라 좌표 재현이 용이했고, 이 때문에 uncertain 없이 전건 confirm 가능했다.
- `content/`에 동등물 부재 → 충돌 위험 없음.
- 구조: `{imperial, alliance}.{fleet[], patrol[], ground[]}`
  - fleet은 `unit`키 + 병합셀 페어(1&2=ヴァルハラ/オーディン … 12=イゼルローン), 동맹 バーラト만 1&2 병합.
  - patrol/ground는 `{from, to}` 범위.
  - **제국 patrol 第48 gap 보존**(47-47 → 49-49, source gap), **第53~60은 system/planet null + `note:"blank in manual"`**.
  - 요새는 `planet_type:"fortress"`(ガイエスブルク/イゼルローン) — fortresses.json 정합.
- 단, フレイア/レンテンベルク cross-source 불일치(배치표 명시 vs galaxy.json フレイア 행성목록 부재)는 `docs/logh7-manual-canon.md`에 known mismatch로 기록 후 인코딩(값은 인쇄 그대로).

## (e) 종합 권고

1. **즉시 반영(ship-stats.json)**: B-1 confirm-high 17필드 그룹 + B-2 confirm-med 4필드(note 명시, med 유지). 단일추출 신뢰가 아닌 **2회 독립추출 합치**가 격상 근거.
2. **현행 유지(변경 금지)**: #8 공작함 전체, #11 양륙함 速度 — refute(변형열 오치환). 현행 json이 이미 標準.
3. **대기**: #3 전함 ビーム 64 — uncertain, 라이브 RE(0x030b/0x0323 소비처)로 base 캐논값 확정 후 적용.
4. **신규파일**: `content/initial-deployment.json` 생성 권고(5/5 confirm-high).
5. 변형열 차용 필드(#3 修理, #6 ビーム/対空/修理)는 `_raw.note`에 base행 공란 사실을 명시하고 confidence high 금지(med 상한).
