# 0x0325 디스패치 스킵 — 실측 중재 (정적 RE + 서버 바이트 캡처)

정본 EXE: `artifacts/logh7-install/…/exe/g7mtclient.exe` (sha256 9c97de2a…). ImageBase 0x400000.
디컴파일: `.omo/ghidra/export/decompiled/g7mtclient.exe_decompiled.c`.
목적(방어적): 2008 종료 MMO "은하영웅전설 VII" 합법 보존 복원 — 자체 서버 0x0325 유닛 레코드를 원본 클라가 파싱·적재하게 한다.

## TL;DR — 근본원인은 "프레이밍/길이/서브헤더 오류가 아니다"

- **서버 0x0325 프레임은 실측상 완전히 정상**이다(아래 §1 round-trip 증명). 길이 프리픽스·체크섬·서브헤더·정렬 전부 맞다.
- **0x0325(52804B body)는 월드진입 8종 중 유일하게 디스패처에 도달하지 못하는 메시지**다(§2 라이브). 스킵은 크기와 **상관**되나(가장 큰 메시지만 실패) 크기를 막는 메커니즘은 없다(§4).
- 클라의 프레임 처리(FUN_006130a0)에서 디스패처 진입 전 사일런트 드롭 후보는 child-codec 복호 반환값 체크(L296148)뿐인데, **그 복호 출력버퍼 용량(codec+0x1c)은 실바이트 디스어셈블로 0xf00e(61454)로 확정** — 52832B 프레임을 여유롭게 수용한다(§4). 오버플로 드롭 불가.
- **게임 수신 경로 전체에 52830 이하 사이즈캡·분할 메커니즘이 존재하지 않는다**(정본 EXE 실바이트 증명, §4). 모든 버퍼 ≥61442, `size over` 게이트는 게임 커넥션이 아님, 길이 프리픽스는 unsigned. **0x31은 키셋업 프렐류드지 대용량 분할이 아니다** → 서버는 0x0325를 쪼갤 수 없고, 쪼갤 필요도 없다. 클라는 단일 프레임 52804B 수신 능력이 있다.
- **결론: 근본원인은 크기/프레이밍/버퍼/분할이 아니다.** 서버 프레이밍(단일 0x0030 프레임)이 정답이고 바꾸면 안 된다. 라이브 스킵은 크기와 **상관**은 있으나(가장 큰 메시지만 실패) 크기를 막는 **메커니즘이 없다** → 남은 원인은 비-크기(전송 타이밍/클라 상태 의존 무시/대용량에서만 갈리는 상류 파싱). **라이브 이분법 훅 1개로 확정**(§6). 근거 없는 프레이밍/버퍼 수정 금물.

## §1. 실제 0x0325 레코드 바이트 + 서버 프레임은 정상 (오프라인 실측)

서버 실 빌더(`buildInformationUnitInner`) + 실 봉투/암호화 경로(`build0030Body` → `pad8` → `encryptBuffer` → `frame0030WithSubheader(sh=4)`)로 생성한 실측:

| 항목 | 0x0325 (fleets) | 0x0323 (성공, 대조) |
|---|---|---|
| inner (msg32) 첫 16B | `00 00 00 00 03 25 01 00 00 00 00 00 00 07 00 01` | `00 00 00 00 03 23 00 00 00 07 …` |
| msg32 dword0 (LE) | 0 | 0 |
| code @4 (BE) | 0x0325 | 0x0323 |
| body @6 첫 8B | `01 00 00 00 00 00 00 07` (count=1 LE, unit[0].id=7 BE) | `00 00 00 07 …` |
| inner.length | 52810 (0xce4a) | 730 (0x2da) |
| 0030 body (pad8 전) | 52818, `%8=2` | 738, `%8=2` |
| enc(padded) | **52824** | 744 |
| TCP 프레임 len 프리픽스 (u16BE@0) | **52830 (0xCE5E)** — u16 적합 | 750 (0x2ee) |
| 전체 프레임 | 52832 | 752 |
| **round-trip**(unwrap→decrypt→parse0030) | **통과** (복호 body 52824, innerLen 52810, checksum OK, code@4 일치) | 통과 |

결론: 길이필드(TCP len=0xCE5E, innerLen=52810)·체크섬·서브헤더(4)·8B 정렬 모두 정확. child-codec 는 블록독립 **Blowfish-ECB** 라 크기와 무관하게 대칭 복원됨 → **서버 인코딩 버그 가설 반증**. 0x0323 vs 0x0325 프레이밍 diff는 **오직 크기**(52832 vs 752), 구조·엔디안·오프셋 동일.

## §2. 라이브 디스패치 실측 — 0x0325만 유일하게 스킵

`.omo/live-qa/m3-worldenter-diag-20260711-141435/wedisp.jsonl` (디스패처 FUN_004ba2b0 @0x4ba2b0 onEnter 훅, code=[esp+4]):

```
seq1 0x206  seq2 0x204  seq3 0x323  seq4 0x315  seq5 0x313
seq6 0x204  seq7 0xb09  seq8 0x323  seq9 0xb0a  seq10 0x313  seq11 0x315  seq12 0xf03
```

- **0x0325는 12개 디스패치 이벤트 어디에도 없다.** 스폰 버스트가 2회 전송됐고(0x323·0x315·0x313·0x204 각 2회) 0x0325는 두 번 다 스킵.
- **가장 큰 성공 디스패치 = 0x0315 (5004B / 프레임 ~5030).** 0x0325 (52804B / 52832) 실패. 크기 천장은 5KB~52KB 사이.
- **중요: 0xb09·0xb0a가 이제 디스패치된다**(seq7/seq9). 즉 이전 `logh7-0325-loader-gate.md`의 "0x0b0a 미송신이 블로커" 진단은 **폐기(superseded)**. grid 괄호는 배선됐고, 유일한 실블로커는 **0x0325 미디스패치** → 유닛 스테이징 테이블 미충전 → activeCount=0 → 마커클릭 null-deref.

## §3. 클라 수신→디스패처 콜체인 + 유일한 pre-dispatch 드롭 지점

(주소 ImageBase 0x400000, re-analyst RE + 본 분석 교차확인)

1. TCP recv → 링버퍼: `FUN_006152b0` (전용 recv 스레드).
2. 디프레이머(`[u16BE len]` 읽기, unsigned `available>=len` 대기): `FUN_00615ad0`.
3. **0x0030 프레임 코덱 (드롭 지점): `FUN_006130a0` @0x6130a0.**
4. 옵코드 파서 → RecvQue 적재: `FUN_004b8850` @0x4b8850.
5. 큐 드레인 → 디스패처: `FUN_004b8950` → **`FUN_004ba2b0`** @0x4ba2b0 (case 0x325). ← 라이브 훅 위치(미발화).

**FUN_006130a0 실코드 (디컴파일 L296091~):**
- L296122 `if ((ushort)param_2 != 0x30)` — 프레임 code≠0x30 이면 별도 경로.
- L296142 `iStack_8 = piVar7[7]` = **codec+0x1c = 복호 출력버퍼 용량(capacity)**.
- L296143 `piVar1 = piVar7+6` = **codec+0x18 = 복호 출력버퍼 포인터**.
- L296144~147 복호 vtable 호출: 입력=enc, **길이 = `framelen - subheader(4) - 2` = 52824**(0x0325), 출력=codec+0x18, **용량 codec+0x1c 를 참조로 전달**(바운드).
- **L296148 `if (cVar5 == '\0')` → `FUN_00614bb0/FUN_00614b30` → `return 0`. = 프레임 드롭, 디스패처 미도달.** ★ 유일한 pre-dispatch 사일런트 드롭.
- L296156 `if (uVar6 == 0x31)` 분기: **0x31 은 키셋업/ack 프렐류드** — 처리 후 `FUN_006130a0` **재귀 호출**로 다음 프레임을 읽는다(L296164). **대용량 조각 재조립이 아니다.** → 서버는 0x0325 를 여러 프레임으로 쪼갤 수 없다; 클라는 **메시지 1개를 프레임 1개에 통짜로** 기대.
- 비-0x31: 완성 메시지 포인터(`piVar7+9`) 반환 → 디스패치.

**RecvQue 적재 `FUN_004b8850` (L36918~):** 옵코드 크기 룩업 `FUN_004b8b00`(case 0x325 = **0xce44=52804**, 바이너리 전체에서 유일 immediate, 파일오프셋 0xb8d3c) → `malloc(0xce44)` → **고정 52804B 복사**(L36947, `param_2>>2` dword 루프). 즉 **짧은 body 수용 안 함** — 복호버퍼에서 항상 52804B 를 읽는다. 500슬롯 큐(stride 0x14).

## §4. 정적 버퍼/게이트 — 게임 수신경로에 52830 이하 사이즈캡 없음 (실바이트 확정)

코덱/트랜스포트 팩토리 실호출(디컴파일 L27792): `FUN_00612030(0x1f4000, 0x3e8000, 0xf000, 3, …, 0x26, 0x10, 4, 0, 0)`.
→ 코덱 생성자 `FUN_006127d0`. 복호 출력버퍼 용량(codec+0x1c)은 Ghidra가 `unaff_EBP`로 소실했으나 **정본 EXE 실바이트 capstone 디스어셈블로 확정**:
size = `[esp+0x10]` 슬롯(0x612afa) ← 0x612813에서 `edx=[ecx+eax+2]`, ecx=서브헤더 4, eax=align8(0xf000)+8=0xf008 → **size = 4 + 0xf008 + 2 = 0xf00e = 61454**.

| 버퍼/게이트 | 값 | vs 52832 프레임 |
|---|---|---|
| recv 링 / send | 0x1f4000(2MB) / 0x3e8000(4MB) | ≫ |
| 프레임 조립(+0x30) | 0xf000+2 = 61442 | > |
| 트랜스포트 프레임(+0x04) | 0xf000 = 61440 | > |
| **복호 출력버퍼(codec+0x1c)** | **0xf00e = 61454** (실바이트 확정) | **>** (오버플로 불가) |
| len 프리픽스 | u_short unsigned, ntohs, unsigned 비교(FUN_00615ad0 L298655) | 0xCE5E 정상 |

- **`size over` 게이트(FUN_006123d0)는 게임 커넥션이 아니다.** 게임 수신핸들러 `FUN_004add60`(@0x4add60, 팩토리 셋업 직후 27804행 등록)은 **게이트 없는 펌프 FUN_006122c0** 사용. size_over 게이트는 다른 커넥션(FUN_004ac670)용.
- RecvQue-post 호출자 **OnRecv @0x4ae0d0** (직접호출 0x4ae0ff/0x4ae127)도 사이즈 게이트 없음 — code≠0x202/0x204면 바로 `FUN_004b8850(code,buf)`.
- 레코드 스트림(FUN_00610420/06104b0)은 **signed int32** 길이 → 경계는 2GB지 32K 아님.

**결론: 게임 수신경로 어디에도 52830을 막는 사이즈캡·분할·버퍼한계가 없다.** 복호버퍼 61454 > 52824. 클라는 0x0325를 **단일 프레임으로 받을 능력이 있다.** §1(ECB 클린 라운드트립) + 여기(버퍼/게이트 증명) ⇒ 정적으로 이 프레임을 드롭시킬 크기 메커니즘은 없다.

## §5. 0x0323(성공) vs 0x0325(스킵) — 크기 상관, 비-크기 원인

구조·엔디안·오프셋·프레이밍 전부 동일. 유일 차이는 크기(enc 744 vs 52824)지만 **§4로 크기 게이트가 없음이 증명됐다.** 즉 스킵은 크기와 **상관**될 뿐(가장 큰 메시지만 실패) 크기가 **원인 메커니즘은 아니다.** 크기와 상관되면서 크기캡이 아닌 후보(라이브 확인 요망):
- (a) **전송 타이밍**: 서버가 0x0325(52KB, 전송 지연)를 클라의 해당 채널 수신펌프 활성화 **전에** 송신 → 폐기.
- (b) **클라 상태 의존 무시**: 특정 상태(예 grid-init 미완/월드모드 플래그) 아니면 0x0325만 폐기.
- (c) **대용량 상류 파싱 분기**: message32 dword0/체크섬/레코드-이터레이터가 대용량에서만 다르게 해석.

## §6. 막힌 부분 → 라이브 이분법 훅 (qa-marker2와 직렬 조율 필요)

정적으로 크기 원인이 배제됐으므로, 스킵 지점을 상류/하류로 **이분(bisect)** 하는 훅이 필요하다(포트 47900 겹침 → qa-marker2와 직렬).

**1차 훅: `OnRecv` @0x4ae0d0 onEnter, `[esp+4]`(code) 로깅.**
- **0x0325가 여기 뜨면** → 트랜스포트/코덱/체크섬/레코드파싱 전부 정상. 드롭은 **하류**(FUN_004b8850 큐 만원, 또는 FUN_004ba2b0 디스패처 상태게이트/case 스킵). → 디스패처 case 0x325 내부 게이트(전략맵 상태 0x126711, count 등)와 큐 500슬롯 만원 여부 계측.
- **안 뜨면** → **상류**(코덱/체크섬/레코드이터레이터)에서 소실. → `FUN_006130a0` 복호 vtable(+0x18) 반환값 + `FUN_00615ad0` 반환 시 `[ecx+0x1e]`(framelen)로 트랜스포트가 52830을 실제로 뽑는지 확인.

## §7. 서버가 고쳐야 할 지점

**프레이밍은 건드리지 마라.** 단일 0x0030 프레임 · 단일 52804B body 가 정답이다(§4: 클라가 통짜 수신 가능, 분할 시 재조립 로직 없어 깨짐). 서버 인코딩/길이/체크섬/버퍼 수정은 근거 없다 — 반증됨(§1·§4).

실제 수정 지점은 §6 라이브 이분 결과로 확정된다:
- **하류(OnRecv에 뜸)면**: 서버 wire 는 정상. 원인은 클라 디스패처 상태게이트 or 큐 상태 → 서버는 **송신 순서/타이밍**(예: 0x0325 를 특정 상태 이후로, 또는 다른 선행 메시지 뒤로) 조정. 프레임 내용 아님.
- **상류(OnRecv에 안 뜸)면**: 대용량 특유의 상류 파싱 실패 → §5(c) 정밀 계측 후 해당 필드(dword0/체크섬 해석)만 조정.
- **타이밍(§5a)이면**: 서버가 0x0325 를 클라 수신펌프 준비 후 송신하도록 시퀀싱.

현재 서버는 `Buffer.alloc(0xce44)` 로 52804B 전량 송신(`server/src/server/logh7-world-records.mjs` `buildInformationUnitInner`) — **이 크기 자체는 정본이며 문제가 아니다**(§4). body 축소·프레임 분할은 하지 말 것.
