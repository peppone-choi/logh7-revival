## 결론 요약 (server-dev가 먼저 읽을 것)

1. **0x0f06은 응답을 요구하는 요청이다.** 클라 셀렉터가 `0x0f06`을 보낼 때 "기대 응답 = `0x0f07`"을 미결 큐에 등록한다. 서버가 `0x0f07`을 안 돌려주면 이 큐가 영원히 막힌다.
2. **서버가 보내야 할 것: `0x0f07` ResponseInformationMessengerStatus, 바디 정확히 29900바이트(0x74cc) 고정 길이.** 값은 **전부 0으로 채워도 된다** (아래 이유).
3. **바디 내용은 클라가 파싱하지 않는다.** 수신 즉시 호출되는 핸들러 `FUN_005266e0`는 **빈 스텁(`return;`)이다.** 클라는 29900바이트를 정적 버퍼에 복사만 하고 아무 필드도 읽지 않는다. 따라서 필드 레이아웃·엔디안은 **월드 진입 경로에서 관측 불가능**하며, zero-fill이 정확·안전한 값이다.
4. **idle 크래시의 1차 원인은 0x0f06 무응답에 의한 송신 파이프라인 정지로 강하게 지목된다** (직렬 큐 정지 메커니즘 + 과거 라이브 대조실험, 아래 §4). 확정 fix = 0x0f07 응답 구현.

## 2. 응답 레코드 바이트 레이아웃 (0x0f07)

| offset | 크기 | 타입 | 엔디안 | 의미 | 근거 |
|---|---|---|---|---|---|
| 0x0000 | 29900 | u8[29900] | — | **전체 바디. 클라가 파싱하지 않음** | 소비자가 원시 dword 복사(L39773 `for iVar15=0x1d33`) 후 `FUN_005266e0`(빈 스텁) 호출 |

- **필드 세분 레이아웃: unknown — 날조 금지.** 즉시 핸들러가 빈 스텁이므로 클라 코드에서 필드 오프셋/타입/엔디안을 관측할 근거가 없다. 복사 버퍼(client base + 0x44fa1c, 29900B)를 나중에 읽는 메신저 UI 코드가 존재할 수는 있으나(미추적), 월드 진입·idle 경로에서는 아무도 읽지 않는다.
- 참고(비검증): 리셋 전 서버 빌더(`git show 5bd249c:server/src/server/logh7-account.mjs` L249)는 `[u16 count][u32 charId, u32 status]×N` (stride 8, LE) 로 추정 구현했으나 이는 **미검증 추측**이었다. 신규 로그인 플레이어는 연락처가 0이므로 count=0(전부 0)이 정답이며, 엔디안 논쟁은 무의미하다.
- `0x03xx` 레코드는 packed BIG-ENDIAN이었지만, **0x0f07은 클라가 안 읽으므로 엔디안 규칙이 적용되지 않는다.** (추측으로 BE/LE를 단정하지 말 것.)

## 3. 서버가 보내야 하는 것 — 바이트 수준 스펙

- **메시지 코드:** `0x0f07`
- **바디 길이:** 정확히 **29900바이트 (0x74cc)**. 사이저가 `*param_3=0`(고정)으로 이 길이를 강제하므로, 짧거나 길면 프레이밍이 어긋난다.
- **바디 값:** 29900바이트 전부 `0x00` 허용 (권장). 클라는 복사만 하고 파싱하지 않는다.
- **프레이밍:** 기존 0x0f01/0x0f03 등 다른 0x0fxx inner 응답과 동일한 message32 프레이밍. `WORLD_RESPONSE_OBJECT_SIZES[0x0f07] = 0x74cc`와 이미 일치(서버 login-protocol 테이블).
- **송신 시점:** 클라가 0x0f06을 보낸 직후 1회 응답. (0x0f02 grid-init 스폰 뒤 관측된 그 0x0f06.)

server-dev 주의: 이 응답은 "내용 정확도"가 아니라 "도착 여부"가 전부다. 빈 29900B로 파이프라인을 즉시 풀 수 있다.

## 4. idle 크래시 인과 사슬 — 코드 근거

### 4a. 송신 큐 메커니즘 (확정, `FUN_004b78a0` + `FUN_004ba2b0` tail)

클라는 **직렬화된 미결 요청 큐** 하나로 요청을 관리한다:

- 큐 상태: `clientBase + 0x357ec0` = head 인덱스(미결 개수), 엔트리 stride `0xc`(12B), 용량 100.
  엔트리 = `[+0x357ec4] reqCode(u16)`, `[+0x357ec8] expectedRespCode(u32)`, `[+0x357ecc] payload ptr`.
- **셀렉터(FUN_004b78a0) 송신 경로 (L36836-36857):**
  - 큐가 비어 있으면(head==0) 요청을 **즉시 전송**.
  - 이미 미결 요청이 있으면 새 요청을 **큐에 적재만 하고 전송하지 않는다**(head 증가). 100 초과 시 wrap하며 `"MAX_QUE!! Waiting"` 로그.
  - `iVar5`(기대응답)가 `-1`이 아닌 요청 = 응답 대기형. 0x0f06은 `iVar5=0xf07`이므로 **응답 대기형**이다.
- **큐 드레인(FUN_004ba2b0 tail, L41072-41106):** 응답 수신 시 `incomingCode == 큐[i].expectedRespCode`(`+0x357ec8`)인 엔트리를 찾아 제거하고 head를 감소시킨 뒤, **새 head 엔트리의 요청을 전송**한다(L41104: `send(sock, +0x357ec4, +0x357ecc)`).

→ **핵심: 응답 대기형 요청이 미결이면, 그 응답이 올 때까지 이 큐로 나가는 후속 요청이 전부 정지한다.** 0x0f07을 안 주면 0x0f06 엔트리가 head에서 영원히 안 빠지고, 뒤에 쌓이는 요청은 한 건도 송신되지 않는다.

### 4b. 하트비트가 계속 흐른 이유 (부분 확정)

관측 로그에서 0x0300 하트비트는 정지 중에도 응답됐다. 셀렉터의 `param_1=='\0'` 분기(L36831-36835)는 **큐를 거치지 않고 즉시 전송**한다. 하트비트류는 이 즉시 경로로 추정되어, 응답 대기 큐가 막혀도 하트비트 자체는 계속 나간다. (`param_1`의 정확한 의미 = 큐드/즉시 구분자, 호출부 미추적 → MEDIUM.)

### 4c. 백오프→사망 (지목, 추가 probe 필요)

- **확정:** 0x0f06 무응답 → 응답 대기 큐 정지 → 이후 큐형 요청(주기적 정보 폴 등) 전부 미송신.
- **강한 방증 (자연 대조실험):** `logh7-record-map-inventory.md:47` — 2026-06-16 세션은 `0x0f06->0x0f07`이 **완료**되자 클라가 정상적으로 월드 진입 후 `0x0b09, 0x0325, 0x0356, 0x1200, 0x1202, 0x0305...` 연쇄를 진행했다. 0x0f07 있으면 진행, 없으면 정지+사망 — 인과의 직접 증거.
- **미확정:** 하트비트 간격 지수 증가(27→39→52→65s)와 exit `0xCFFFFFFF`가 큐 정지의 직접 결과인지, 별도 idle 워치독인지는 하트비트/idle 스케줄러 코드 미추적으로 단정 못 함. **다음 probe:** 0x0f07 구현 후에도 백오프·사망이 남으면 idle 타이머(하트비트 스케줄러)를 별도 조사. 매달리지 말 것 — 우선 0x0f07을 채우고 재현하라.
