## Server-data boundary

The inventory distinguishes recovered protocol records from authored server seeds.

DB/backend/client handoff rule:

| Boundary | Stable format | Do not confuse with |
| --- | --- | --- |
| Content DB -> backend | SQLite catalog rows from `logh7-content-db.mjs`: systems, planets, fortresses, posts, roster, characters, commands, unit/ship classes, client strings | live character/session/world state |
| Backend -> client | parser-specific wire records built per code: `0x1008` 128-byte packed create OK stream, `0x2004` compact card stream, `0x0323` 724-byte character record, `0x0325` unit table, `0x0313/0x0315` map tables, `0x0321` facility table | raw DB rows or JSON object shapes |
| Client local resources | constmsg/MsgDat/UI strings, portraits, graphics, installed EXE logic | server-owned dynamic data such as current location, rank assignment, office occupant, fief owner, battle damage |

The original server must have supplied dynamic values for current location, starting post, office/room
state, fief ownership, fleet positions, and session progression. The local DB does not prove those values;
when we create them, they are authored server state and must be tagged as reconstructed/provisional.

Use as real or high-confidence server data:

| Surface | Current state |
| --- | --- |
| Lobby/session flow | `0x2004`, `0x2006`, `0x200a` are implemented and live-observed. |
| Character identity records | `0x0323`/`0x034f` layouts are implemented; names still depend on safe provenance. |
