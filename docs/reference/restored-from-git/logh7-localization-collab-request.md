# LOGH VII 한글화 — 협업자(파일 디코드 · 도구 제작) 요청서

작성일: 2026-06-14 · 작성자: Windows 세션(클라이언트/frida 실행 담당) · 수신: iOS/Mac 협업자(오프라인 디코드·도구 담당)

## 0. 30초 요약 (협업자가 할 일)

1. **전체 문자열 인벤토리 + JP→KO 번역 워크시트**를 만든다 (Task A). 게임이 실제 화면에 그리는 모든 문자열을 `constmsg.dat` / `MsgDat/*` / `exe/String.txt`에서 디코드하고, 일본어 원문 ↔ 현재 한국어 ↔ 상태(번역됨/미번역)를 한 표로.
2. **코드페이지 강제 패치 레시피 도구**를 만든다 (Task B). 이번에 확정된 사실 = 클라가 텍스트를 **코드페이지 932(Shift-JIS)** 로 변환한다 → 이걸 **949(CP949)** 로 강제하면 번역 문자열이 100% 정상 변환됨(아래 §3 증거). 같은-길이 바이트 패치 JSON을 산출하는 크로스플랫폼 Python 도구.
3. **"글자 안보임" 렌더링 미해결**을 조사한다 (Task C). 변환은 풀렸지만 글리프가 실제로 그려지는지가 아직 미확정. 화면캡처에 의존하지 않는 결정적 검증 수단이 필요.

> **핵심 제약**: 협업자는 Windows 게임 클라이언트/frida/pywin32를 **실행하지 않는다**고 가정한다. 모든 산출물은 **순수 파일 디코드 + 표준 라이브러리 Python(크로스플랫폼)** 으로 동작해야 한다. Windows에서만 가능한 검증(라이브 클라 실행)은 이 Windows 세션이 맡으니, 협업자는 그쪽이 실행할 수 있도록 **사양(spec)** 만 정확히 적어 넘기면 된다.

## 1. 배경과 "방법 변경"

기존 방법은 이 Windows 머신에서 **라이브 frida 캡처 + 추측 바이너리 패치**를 반복하는 것이었다. 이 방식은 (a) 게임이 풀스크린/창모드를 오가며 화면을 점유하고, (b) D3D8 표면이라 스크린샷 검증이 불안정하며, (c) 패치 후보를 한 번에 하나씩만 실험할 수 있어 느리다.

**바뀐 방법**: 디코드/번역/도구 제작처럼 **오프라인·결정적·반복가능**한 작업은 협업자에게 위임한다. 이 Windows 세션은 협업자가 만든 레시피/도구를 받아 **라이브 클라이언트로 최종 검증**만 수행한다. 즉 "실험을 화면 앞에서 추측"하던 걸 "사양을 파일로 확정 → 1회 검증"으로 바꾼다.

## 2. 협업자 환경 제약 (반드시 지킬 것)

- **OS 독립**: macOS에서 동작해야 한다. `pywin32`, `frida`, `ctypes.windll`, Windows 경로 하드코딩 금지.
- **Python 표준 라이브러리 우선**. 외부 의존이 필요하면 `pyproject`/`requirements`에 명시하고 최소화. 기존 저장소 도구들은 대부분 stdlib-only다(예: `tools/standalone/logh7_face_registry.py`).
- **바이너리 입력은 저장소 안 파일**로 한정. 클라이언트 EXE/데이터는 이미 추출되어 있다(아래 §7 경로). 협업자가 별도 CD/ISO를 다룰 필요 없음.
- **인코딩 규칙**: 번역 카탈로그/원문은 **UTF-8**로 관리, 클라가 읽는 산출 파일은 **CP949 바이트**로 생성(아래 §8). 길이/오프셋 테이블을 깨면 안 됨.
- **출력은 결정적**: 같은 입력 → 같은 출력(해시 안정). 타임스탬프/난수로 흔들리지 않게.

## 3. 이번 세션에서 **확정된 사실** (frida 라이브 캡처)

도구 `tools/logh7_frida_mbtowc.py`(frida 17.x API 수정 + `--force-cp` 모드 추가 완료)로 KO 오버레이를 올린 실클라의 `kernel32!MultiByteToWideChar` 호출 215건을 캡처했다. 증거 원본: **`.omo/ui-explorer/mbtowc-capture-949.json`**.

| 항목 | 값 | 의미 |
|---|---|---|
| 코드페이지 | **932 (Shift-JIS)** — 215건 전부 | 이전 진단 "65001(UTF-8 ACP)"은 **오진**. 실제는 932. 시스템 ACP와 무관하게 클라가 932를 쓴다. |
| dwFlags | **9 = `MB_PRECOMPOSED(1) | MB_ERR_INVALID_CHARS(8)`** | 잘못된 바이트열이면 변환이 **실패(반환 0)** → 화면 공백. |
| cbMultiByte | **-1** | 입력은 NUL 종단 문자열. |
| 949 강제 결과 | 번역된 CP949 문자열 = **정상 변환(ret>0)**, 미번역 일본어 SJIS = **실패(ret=0) 또는 깨짐** | **수정 = 변환 코드페이지 932→949 강제**가 정답임을 증명. |

**949 강제 시 정상 변환된 한국어(원문 jp가 반각가나 깨짐으로 나오는 = 진짜 우리 CP949 번역)**:

```
게임 시작(6)  새 캐릭터 작성(9)  오리지널 캐릭터 추첨(12)  캐릭터 삭제(7)
세션 변경(6)  환경 설정(6)  크레딧(4)  게임 종료(6)  서버 공지(6)  종료(3)
로그인(4)  비밀번호(5)  ID와 비밀번호를 입력하세요(17)  로그인할 캐릭터를 선택…(17)
로그인에 실패했습니다.(13)  버전이 다릅니다.(10)  서버에 접속합니다.(11)
캐릭터 정보(7)  이름을 입력하세요.(11)  이름(3)  뒤로(3)  취소(3)  등록(3)
오리지널 캐릭터 선택(12)  플레이 가능 캐릭터(11)
```

→ **타이틀 메뉴·로그인·캐릭터 생성 UI의 번역분이 코드페이지만 949면 완전히 정상 변환된다.** 변환·폰트로드·텍스트로드는 정상. 유일한 변환 측 벽은 **코드페이지 932**였고, 이건 강제로 해결 가능.

### 왜 과거 "push 0x3b5" 패치는 공백이 됐나 (해소된 미스터리)

정적 워크플로우(`wf_eb82a9c6-06a`)가 바이트 단위로 규명했다. **두 가지 원인의 합**이다:
1. **커버리지**: MBToWC는 단일 IAT 슬롯 **`0x66b170`(file `0x26b170`, import hint 0x1e4)** 을 **22개 콜사이트**가 호출한다(`call dword[0x66b170]` = `ff 15 70 b1 66 00`). 과거 패치는 그중 **3곳**(`FUN_00600394`)만 바꿔, 메뉴 렌더 경로의 다른 변환은 그대로 932로 남았다.
2. **strict 플래그**: `FUN_00600394`의 사이징/복구 패스는 **`MB_ERR_INVALID_CHARS(0x08)`** 포함 플래그 `0x09`를 쓴다(사이트 #1 `0x6003f5`, #3 `0x60047e`; 메인 변환 #2 `0x600454`만 `0x01`). cp를 949로 강제해도 미번역/잘못된 바이트가 들어오면 strict 패스가 **ret=0(하드 실패)** → 길이 0 → CRT 포매터가 **빈 문자열** 산출 → 공백. (원본 65001/932에선 같은 바이트가 U+FFFD 박스=모지바케로 나옴 → 그래서 "박스→공백"으로 바뀐 것.)

> **라이브 캡처 교차검증(중요)**: 메뉴 시점 글로벌 코드페이지는 **이미 932**(setlocale("Japanese") 경로). 따라서 "생산자에서 글로벌을 949로 강제"하는 후보(`FUN_005fff07`/`0x609c5d`)는 **글로벌이 0일 때만 실행되는 가드**에 막혀 **런타임에 SKIP**된다. **보장 경로 = IAT 슬롯 `0x66b170`을 트램폴린으로 가로채 cp 932→949 강제** — 이건 라이브 frida `--force-cp 949`가 이미 한국어 정상 변환(ret>0)으로 증명했다(Task B).

## 4. 미해결 문제 #1 — "글자 안보임" (렌더링)

변환(코드페이지)은 풀렸지만, **글리프가 실제로 화면에 그려지는지**는 아직 미확정이다. 이번 캡처 중 스크린샷(`.omo/ui-explorer/menu-949-{a,b}.png`)은 게임이 작은 **창모드**로 떴고 GDI `ImageGrab`이 **D3D8 표면을 못 잡아** 비어 보였다(=캡처 한계일 수 있음, 실모니터엔 떠 있었을 수 있음). 그러나 "글자 안보임"을 **독립 미해결 문제**로 확정하고 결정적으로 검증해야 한다.

분리해서 봐야 할 두 단계:
- **(1) 변환** ANSI(932/949)→UTF-16: **해결**(§3).
- **(2) 렌더링** UTF-16/ANSI → GDI 텍스트 출력(`ExtTextOutA`/`TextOutA`/`DrawTextA`, `CreateFontA` charset=HANGEUL(0x81), 폰트명 `굴림`): **미확정**.

(2)의 의심 지점: 굴림 폰트 미설치/이름불일치, charset과 코드페이지 불일치, D3D 텍스처폰트 경로(`FUN_004eb100`은 이미 u16 wide를 읽음 → 상류 변환이 진짜 입력)에서 폭 측정/클리핑으로 0폭 처리, 또는 단순 캡처 한계. → **Task C**에서 화면캡처에 의존하지 않는 검증 수단을 만든다.

## 5. 요청 작업 (Deliverables)

### Task A — 전체 문자열 인벤토리 + JP→KO 번역 워크시트  **[최우선]**

**목표**: 게임이 화면에 그리는 **모든 문자열**을 한 곳에 모아, 일본어 원문 ↔ 현재 한국어 ↔ 상태로 정리. 이번 캡처가 증명하듯 환경설정·캐릭터생성·능력치 화면 등 **상당량이 아직 미번역 일본어**(§6).

**입력 파일**:
- `.omo/work/logh7-installed/data/MsgDat/constmsg.dat` (HFWR 오프셋테이블, ~115KB)
- `.omo/work/logh7-installed/data/MsgDat/` 하위 `messages_*.dat` 류
- `.omo/work/logh7-installed/exe/String.txt`
- 기존 카탈로그: `content/localization/constmsg-ko.json`, `content/client/{msgdat,schema,message-tokens}.json`

**도구**: 기존 `tools/logh7_msgdat.py`(디코더), `tools/logh7_msgdat_encode.py`(HFWR 재인코더)를 확장/재사용. **macOS에서 동작 확인** 필요(현재 stdlib 기반인지 점검하고 아니면 stdlib로 정리).

**산출물**:
- `content/localization/strings-worksheet.json` (또는 `.tsv`): 행 = `{ source: "constmsg|String.txt|messages", id/offset, jp(SJIS 원문 UTF-8 디코드), ko(현재값 or null), status: translated|untranslated|partial, max_byte_len(CP949 한도) }`.
- 사람이 채우기 쉬운 형태(TSV 권장). 미번역 행이 명확히 보이게.

**수용 기준**: 캡처(§6)에 나온 미번역 일본어 문자열이 워크시트에서 `untranslated`로 식별되고, 각 행에 **CP949 재인코딩 시 허용 바이트 길이**(원본 슬롯 길이)가 표기됨.

### Task B — 코드페이지 932→949 강제 패치 레시피 도구  **[최우선]**

**목표**: 클라가 텍스트 변환에 쓰는 코드페이지 글로벌을 **949로 강제**하는 **같은-길이 바이트 패치 레시피(JSON)** 를 산출. 이 JSON은 이 Windows 세션이 `tools/logh7_codepage_patch.py`(드리프트 검증 동일-길이 패처)로 적용 후 라이브 검증한다.

**검증된 패치 설계 (정적 워크플로우 + 라이브 캡처 교차검증 완료)**. 클라 EXE: `.omo/work/logh7-installed/exe/G7MTClient.exe` (imagebase `0x00400000`, .text `file_off = VA - 0x400000`, 순정 SHA head `2848be76`). 우선순위 순:

- **① (권장·보장) IAT 트램폴린** — cp 932→949를 kernel32 경계에서 강제. 라이브 frida `--force-cp 949`가 정확히 이 동작으로 한국어 정상 변환(ret>0)을 **이미 증명**했다.
  - MBToWC IAT 슬롯 = **VA `0x66b170` / file `0x26b170`** (import hint 0x1e4). 22개 콜사이트가 전부 `call dword[0x66b170]`(`ff 15 70 b1 66 00`). 슬롯은 로더가 런타임 resolve(unbound, TimeStamp=0)라 **슬롯 값 자체는 패치 불가** → 콜사이트 disp32를 재타깃.
  - 코드케이브 = **VA `0x66acd5` / file `0x26acd5`** (.text 끝 811B 제로 영역 중 미참조 0xab 구간). 여기에 **섀도우 슬롯 셀 + 45B 트램폴린**을 쓴다. 트램폴린: 인자 cp(esp+4)∈{0,932,65001}이면 949로 치환 → 진짜 `[0x66b170]`로 jmp.
  - 22개 콜사이트의 disp32를 `0x66b170`→`0x66acd5`(섀도우 셀)로 재기록(같은 길이). 도구: `tools/logh7_x86_patch.X86Builder`(다른 길이/케이브).
- **② (동반·같은 길이) strict 플래그 완화** `MB_ERR_INVALID_CHARS(0x08)` 제거: `6a 09`→`6a 01` at **VA `0x6003f5`(file `0x2003f3`)** 와 **VA `0x60047e`(file `0x20047c`)**. 효과: 미번역/잘못된 바이트가 **전체 공백(blank) 대신 U+FFFD**로 degrade. (Site2 `0x600454`는 이미 `0x01`, 손대지 말 것.) 같은-길이라 `tools/logh7_codepage_patch.py`로 적용 가능.
- **③ (대개 무효 — 참고만) 생산자 DAT-글로벌 강제** at VA `0x609c5d`(file `0x209c5d`): `0fb705ac063503687406350368041000005053e86b3d000083c410`(27B) → `c70574063503b503000033c0`+`90`×15(`mov dword[0x03350674],0x3b5; xor eax,eax; nop…`). **단 가드 `cmp [0x3350674],0; jnz`(VA `0x609c5b`)가 글로벌이 0일 때만 이 경로를 타므로, 라이브상 메뉴 시점 글로벌은 이미 932(setlocale) → 런타임 SKIP. 가드 무력화를 동반하지 않으면 효과 없음.** ①을 우선.

**도구 사양 (크로스플랫폼 Python, EXE 바이트만 읽음 — macOS 가능)**:
- 입력: EXE 경로.
- PE 파싱은 stdlib로(또는 저장소의 `tools/logh7_child_codec._parse_pe_image` / `_virtual_address_to_offset` 재사용 — 순수 Python인지 확인).
- 출력: 같은-길이 패치는 `{"patches":[{"va","originalHex","patchedHex","note"}]}` → `tools/logh7_codepage_patch.py` 포맷. 케이브/IAT 패치는 `tools/logh7_x86_patch.X86Builder` 레시피.
- **산출 대상 = 위 ①(IAT 트램폴린) + ②(strict 플래그 완화)** 를 한 번에 적용하는 빌더. ①이 보장 경로(라이브 증명), ②는 미번역 잔여 텍스트의 graceful degrade. ③(생산자 강제)은 가드 SKIP이라 기본 비활성(필요 시 가드 무력화 옵션). 모든 `originalHex`는 순정 EXE에서 직접 읽어 드리프트 가드 통과시킬 것.

**참고**: 이 저장소에서 **백그라운드 정적분석 워크플로우**가 같은 패치를 설계·검증 중이다(런 ID `wf_eb82a9c6-06a`). 결과가 나오면 정확한 오프셋/원본바이트가 이 문서 또는 `.omo/`에 갱신된다 — 협업자는 그 검증된 오프셋을 받아 도구화하면 된다. 그 전엔 위 VA를 EXE에서 직접 검증해 사용.

**수용 기준**: 산출 JSON을 `logh7_codepage_patch.py`로 적용했을 때 드리프트 에러 없이 통과(원본바이트 일치), 패치된 바이트가 디스어셈블 상 "글로벌=949"를 실제로 달성. (최종 화면 검증은 Windows 세션이 수행.)

### Task C — "글자 안보임" 렌더링 검증 (화면캡처 비의존)  **[중요]**

**목표**: 변환이 성공해도 글리프가 안 그려지는지/캡처 한계인지 **결정적으로** 가른다.

**오프라인(협업자가 직접 가능한 부분)**:
- 굴림+`HANGEUL_CHARSET` 가정의 **폰트/charset 정합성 감사**: 폰트 패치 도구 `tools/logh7_japanese_font_patch.py`의 charset/폰트명 바이트가 EXE에서 올바른지 정적 검증(2곳). 폰트명 `굴림`(`0x36e240) 패치 위치/바이트 확인.
- **재현 가능한 렌더 단위 테스트(스크린샷 없이)**: CP949 문자열 → UTF-16(코드페이지 949) → (가능하면 PIL로 굴림류 글리프 존재 여부) 렌더하여 글리프 누락/0폭을 표 형태로 보고. 최소한 "이 코드포인트들이 굴림에 존재하는가"를 데이터로 산출.

**Windows 세션이 실행할 사양(협업자가 spec만 작성)**:
- `ExtTextOutA`/`TextOutA`/`DrawTextA` 후킹으로 **실제 그려지는 (hwnd, hFont, lpString 바이트, nCount, 반환폭)** 을 로깅하는 frida 스크립트 사양. 입력 코드페이지 949 강제 상태에서 텍스트 출력 콜이 **호출되는지/문자열이 올바른지/그려지는 폭이 0인지**를 확정. (구현은 Windows 세션, 사양·기대값 표는 협업자.)

**수용 기준**: "글자 안보임"의 원인이 (a)캡처 한계 (b)폰트 글리프 부재 (c)렌더 콜 미발생/0폭 중 **무엇인지** 데이터로 좁혀지는 산출물(표/JSON).
