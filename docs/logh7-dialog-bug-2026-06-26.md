# char-creation 확인 다이얼로그(예/아니오) 무반응 — deep-RE + 수정 제안 (2026-06-26)

대상: real-login(10차 run) 캐릭생성 후 "정말…하시겠습니까?" 예/아니오 다이얼로그가
**클릭·키보드(hw Enter/Space) 둘 다 무반응 → 월드 미진입**. autologin은 이 다이얼로그를 우회(월드 직행).
메모리 "예아니오 다이얼로그 EXE버그".

기준 EXE: `.omo/ghidra/bin/G7MTClient.exe` (RE 인덱스 baseline). 도구: `tools/logh7_redex.py`,
capstone/pefile 직접 디스어셈블.

## 1. 다이얼로그 처리 함수 체인 (RE 확정)

| 함수 | 역할 |
|---|---|
| `FUN_0056ebf0` | **제네릭 다이얼로그 위젯 빌더**. window/dialog_par 아틀라스로 위젯 0~0xb 레코드 생성. 버튼6/7 = constmsg **group 0x67**(=login-error 문자열, dlgfix가 라벨만 교정) push. 끝에서 `FUN_00507a50` + `FUN_00570340(this, 1)` 호출. |
| `FUN_00570340(this, page)` | **다이얼로그 페이지(입력-활성 그룹) 활성기**. 정적 테이블 `DAT_00675138`(page*0xa0 stride, 엔트리 8B, 페이지당 0x14) 참조. page의 그룹 목록에 든 위젯만 `FUN_00502eb0/ee0/5024e0(w,1)`로 **표시+enable+클릭가능**화, 나머지는 (,0). **활성 성공 시 `*(this+0xde0)=1` 세팅.** page의 첫 엔트리가 -1이면(빈 페이지) `FUN_00502ea0(0);FUN_005024b0(0)`(입력 전면 차단) 분기 → **+0xde0 미세팅**. |
| `FUN_004fdde0(this, idx, page)` | __thiscall(ret 8). 캐릭생성 루트에서 확인 다이얼로그를 idx번째 서브객체(`this+idx*0xde4+0x1948`)로 열며 `FUN_00570340(sub, page)` 호출. |
| `FUN_0054a3d0` | **캐릭생성 최종 확인 소비자**. 두 지점에서 다이얼로그를 연다(아래 ②). 결과 `+0xde0`==3(예)/4(아니오) 판정. |
| `FUN_0054eda0` | 다이얼로그 opener(step→page 매핑 `local_18[param_2*3]` = page 5 또는 4). `FUN_004b68f0` else(메뉴-모드) 분기에서 매프레임 호출. |
| `FUN_0054ee60` | 다이얼로그 결과 poller. `FUN_0056f960`(입력 디스패치) 호출 후 `+0xde0`==3/4면 다이얼로그 닫고(`FUN_00570340(,1)`) 3/4 반환. |
| `FUN_0056f960` | **★실제 클릭/키 입력 디스패치(핵심)**. 예 버튼=`this+0x24`, 아니오 버튼=`this+0x28` 위젯. `FUN_005015f0(2,widget,…)`(클릭-edge)로 눌림 검출 → 예면 `+0xde0=3`, 아니오면 `+0xde0=4`. |

활성 페이지 테이블(EXE에서 직접 덤프, `DAT_00675138`):
```
page 1: []            ← 빈 페이지(입력 전면 차단)
page 2: [3,6,7,4,9]
page 3: [3,6,7,4,5,12]
page 4: [3,6,7,4,5]    ← 예/아니오(group 6,7) 포함
page 5: [2,6,4,5]      ← 예/아니오 포함
```

## 2. 버튼 무반응 근본원인 (RE)

`FUN_0056f960`(입력 디스패치) **상단 게이트가 결정적**:
```
0x56f998  mov  eax,[esi+0xde0]
0x56f99e  cmp  eax,1
0x56f9a7  je   0x56f9b2
0x56f9a9  cmp  eax,2
0x56f9ac  jne  0x56fb04      ← +0xde0 ∉ {1,2} 이면 모든 버튼 hit-test/키 처리 SKIP(페이드 tail로 직행)
```
즉 **예/아니오 버튼은 `+0xde0`(다이얼로그 상태)가 1 또는 2일 때만** 폴링된다. `+0xde0=1`은
`FUN_00570340`이 **유효(비어있지 않은) 페이지로 활성에 성공할 때만** 세팅한다.

→ 무반응의 직접 기전 = **다이얼로그가 빈 page 1로 활성된 상태**(또는 활성 자체 미수행).
page 1은 그룹 목록이 비어 `FUN_00570340`이 입력-차단 분기를 타고 `+0xde0`를 1로 올리지 못한다.
그 결과 `FUN_0056f960`의 상단 게이트가 닫혀 클릭·키 둘 다 콜백(`+0xde0=3/4`)을 못 만든다.
이는 라이브 증상(클릭 좌표 다수·hw Enter/Space 전부 무반응)과 정확히 일치한다.

★단, **정적 경로는 그 자체로 정합**하다:
- `FUN_0054a3d0`의 두 open 지점은 **유효 페이지**를 넘긴다 — 디스어셈블 확정:
  - `0x54a44c: push 4; push 3; mov ecx,0xc9e638; call FUN_004fdde0` → idx=3, **page=4**
  - `0x54a594: push 5; push 3; …` → idx=3, **page=5**
  (디컴파일의 `FUN_004fdde0(3,4)` 2-인자 표기는 정확. page 인자 누락 아님.)
- page 4/5 모두 group 6,7 포함 → 위젯 enable + `+0xde0=1` 세팅 → 입력 게이트 OPEN.
- poller `FUN_0054ee60`은 `FUN_004b68f0`의 **else(메뉴-모드) 분기**에서 매프레임 디스패치됨.

따라서 **정적으로는 다이얼로그가 작동해야 한다.** 라이브 무반응은 **런타임 상태 게이트**에서
온다 — 가장 유력한 둘:
 (a) 캐릭생성 루트가 `FUN_0054a3d0`의 open 지점에 도달하지 못해(스텝 플래그
 `param_1[0x358374..0x358382]` 상태 머신 경로 미충족) 다이얼로그가 page 4/5가 아닌 잔류 page 1로 남음, 또는
 (b) `FUN_004b68f0` 모드 분기가 메뉴-모드 else가 아닌 strategic 분기(`FUN_004fef90`/`FUN_0050d230`,
 C002 mode2/mode0 경로)로 가서 poller `FUN_0054ee60` 자체가 호출되지 않음.
C002 전례(정적 객체/모드 식별이 라이브에서 수차 반증됨)상 (a)/(b)는 **라이브 probe로만 확정** 가능.

## 3. menufix 동형 여부 — **아니오(다른 메커니즘)**

- **menufix**(`FUN_0051a370` @ 0x51ab35): 로컬 enable 배열 `[1,0,0,0,0,1,0,1]`을 루프로
  `FUN_00502780(1,idx,flag)`+`FUN_005024e0`에 먹여 8개 로비 버튼을 개별 enable. 패치=imm8 0→1.
- **다이얼로그**: enable 배열 패턴을 **쓰지 않는다.** 클릭가능성은 `FUN_00570340`의 **페이지-그룹
  테이블**과 `FUN_0056f960`의 **`+0xde0` 상태 게이트**가 지배.
→ menufix의 "imm8 enable 0→1 되돌림" 벡터와 **동형이 아니다.** 동형 패치 직접 적용 불가.
- **dlgfix**(현 DEFAULT_STACK 포함)는 버튼 **라벨만**(group 0x67→0x62, 로그인실패→결정/취소)
  교정할 뿐 **입력 게이트와 무관** → 이번 무반응 버그를 고치지 않는다(혼동 주의).

## 4. 수정 제안 (RE-확정 근거 / same-length / originalHex 가드)

추측 P0 패치 금지 원칙에 따라, **근본 게이트 강제(positive-control) 후보**만 제시하고
라이브로 어느 게이트가 닫혀 있는지 확정한 뒤 최소 패치를 고정한다.

### 후보 A — 입력 디스패치 상태 게이트 무력화 (가장 직접적, 진단 겸용)
`FUN_0056f960`의 `+0xde0 ∉ {1,2}` skip을 제거해, page와 무관하게 예/아니오 위젯을 항상 폴링.
```
VA 0x0056f9ac  originalHex: 0f 85 52 01 00 00   (jne 0x56fb04)
               patchedHex : 90 90 90 90 90 90   (6×NOP, same-length)
```
주의: `+0xde0`는 "이미 눌림" 상태도 인코딩하므로 NOP만으로는 더블-트리거 위험. **진단용으로 먼저**
적용해 "버튼이 살아나는가"만 본 뒤, 살아나면 후보 B로 좁힌다.

### 후보 B — 다이얼로그 활성 페이지 강제 (page 1 잔류가 원인일 때)
빌더 말미 `FUN_0056ebf0` 기본 활성을 page 1 → page 4(예/아니오 포함)로 변경.
```
VA 0x0056f8c0  originalHex: 6a 01   (push 1)
               patchedHex : 6a 04   (push 4, same-length)
```
이는 빌더 기본값만 바꾸므로, 정상 open 경로(page 4/5 명시)에는 영향이 작고 page 1 잔류 케이스를 구제.
(부작용: page 1을 의도적으로 쓰는 다른 다이얼로그가 있으면 회귀 → 라이브에서 로비/세션 다이얼로그
동시 확인 필수.)

### 권장 순서
1) 후보 A를 **읽기전용 진단**으로 적용 → 버튼 반응 여부로 (게이트 원인) vs (poller 미호출) 판별.
   - 버튼 살아나면 = `+0xde0` 게이트(page) 문제 → 후보 B로 교체.
   - 여전히 무반응이면 = `FUN_0054ee60`/`FUN_0056f960` **미호출**(모드 분기 b) → 패치 대상이
     `FUN_004b68f0` 모드 게이트로 이동(C002 mode2/mode0와 동일 전선, 별도 cave 필요).
2) 확정된 단일 게이트만 최소 same-length 패치로 고정. 신규 cave는 안전한 int3 cave(0x5d5290)만 사용.

## 5. byte-verify (기준 EXE에서 확인 완료)

```
0x56f8c0  6a 01 8b cd e8 ...     ← push 1; call FUN_00570340  (후보 B 원본 6a 01 확인)
0x56f9ac  0f 85 52 01 00 00      ← jne 0x56fb04               (후보 A 원본 확인)
0x54a44c  6a 04 6a 03 b9 38 e6 c9 00 e8 ...  ← open#1 page=4   (정상 경로 확인)
0x54a594  6a 05 6a 03 b9 38 e6 c9 00 e8 ...  ← open#2 page=5   (정상 경로 확인)
0x56f998  8b 86 e0 0d 00 00 / 83 f8 01 / 74 / 83 f8 02 / 0f 85 ← +0xde0 게이트 확인
DAT_00675138 page1=[] page4=[3,6,7,4,5] page5=[2,6,4,5]        ← 페이지 테이블 확인
```
후보 패치는 모두 same-length, originalHex 16B 미만 단일 명령 시그니처로 가드 가능.
빌드는 `tools/logh7_build_playable_client.py --patches <stack> dialog-inputgate`(신규 JSON
`tools/client_patches/dialog-inputgate.json`)로 캐논 미교체 후보 빌드 권장.

## 6. 라이브 검증 절차

1. stop(이전 node/클라 종료, SHA 복원) → 후보 EXE 빌드(캐논 미교체, 별도 파일명).
2. `ui_explorer --display-mode windowed login --account <id> --password-stdin`(run_login_flow,
   **autologin 금지=실수동 로그인**, 메모리 원칙). 0x7000→0x0020 로비→0x2005 확인.
3. 캐릭생성(초상화 1 + 이름 입력, 예: "Reinhard Lohengram"). 창모드→풀스크린 자동전환 확인.
4. 8단계 폼 완주 → **확인 다이얼로그 출현** 시 trace 켜고 `FUN_0056f960` 진입/`+0xde0` 값 probe.
5. 예(결정) 버튼 좌표 클릭 + hw Enter. **기대: `+0xde0`가 3으로 전이 → `FUN_0054ee60`가 3 반환
   → 0x1008 캐릭확정 송신 → 0x0f02 월드진입.**
6. 결과를 `docs/logh7-client-state-journal.md`에 전진/정체/회귀로 기록(매 테스트 기록 원칙).
7. 회귀(로비/세션 다이얼로그 무반응) 점검 후에만 캐논 스택 편입.
이게 풀리면 real-login→월드 도달 → own-fleet C002 가설(서버 0x0b07 푸시/mode2-mode0) 라이브 검증 가능.

## 7. 잔여 불확실 (정직 정정)

- **정적 경로는 정합**(page 4/5, +0xde0=1, poller 메뉴-모드 디스패치) → 라이브 무반응의 **정확한
  닫힌 게이트는 미확정**. (a)다이얼로그 page 1 잔류 vs (b)poller 미호출(모드 분기) 둘 중 하나로
  좁혀졌고, **라이브 probe 1회**(후보 A 진단)로 판별 가능.
- 후보 B(page 1→4 강제)는 page 1을 의도적으로 쓰는 다른 다이얼로그가 있으면 회귀 위험 → 라이브
  로비/세션 다이얼로그 동시 확인 필요.
- (b)로 판명 시 패치 전선이 `FUN_004b68f0` 모드 게이트(C002 mode2/mode0)로 이동 → same-length로
  안 끝나고 cave detour 필요할 수 있음.
- C002 전례상 정적 객체/모드 식별이 라이브에서 반증된 사례 다수 → 본 RE도 라이브 확정 전까지 P1.
