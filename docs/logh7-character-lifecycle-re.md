## 1. 생성 폼 규칙 (커스텀 0x1008) — 확신도: 高 (메모리 구조체 필드 write + 사이저)

커스텀 생성 위저드는 로비 시퀀서 `FUN_0051a370`의 서브위저드 `FUN_00594f20`(상태 0x40~0x69 점프테이블)로 구동된다. 위저드 상태 맵(점프테이블 0x595bd8/인덱스 0x595c4c 전수 디코드):

|---|---|---|---|
| 0x40 | `FUN_00595c80` | 2 | 원작(item2) 진입 → 0x1000 송신 |
| **0x41** | **`FUN_00595d90`** | 0x14 | **커스텀 생성 진입 → 0x0203 SelfInfo 송신** |
| 0x54 | `FUN_00595db0`+`FUN_00595f00` | 0x15 | 생성 구조체 초기화(§1.1) |
| 0x55 | `FUN_005960b0` | 0x16/0x39 | 진영(power)·페이지 |

### 1.1 인메모리 생성 구조체 `DAT_02227f60` (고정 0x80=128B, `FUN_00595db0`가 zero-init)

위저드 각 페이지가 이 고정 구조체에 필드를 채우고, 최종 `FUN_00595de0(category)`가 0x1008로 송신한다. **필드 오프셋(구조체 베이스 기준)과 강제 값 범위:**

| struct off | 필드 | 값 범위 (폼 강제) | write 근거 |
|---|---|---|---|
| +0x00 | request_category (u8) | 0/1/2/3/4 (단계, **4=최종 커밋**) | `FUN_00595de0(0..4)` @ 192883/192941/192979/193018/**193125** |
| +0x04 | account_token (u32) | clientbase+0x3584a0 값 | `DAT_02227f64` @ 193295 |
| **+0x08** | **power(진영, u8)** | **{2, 3}** — 2택 라디오 | `DAT_02227f68` @ 193429(=2)/193437(=3) |
| **+0x09** | **blood(출신, u8)** | **power에 종속**: power==2 → {0,1,2,4}; power==3 → {3,4} | `DAT_02227f69` @ 193596~193733 |
| **+0x0a** | **sex(성별, u8)** | **{0, 1}** | `DAT_02227f6a` @ 193502(=0)/193510(=1) |
| +0x0b | lastname_len (u8) | =실자수+1 (NUL 포함), **실자수 ≤ 13** | `DAT_02227f6b` @ 193829; 사이저 `cmp [ecx+0xb],0xd; jbe` @ 0x405745 |
| +0x0c | lastname UTF-16LE | ≤13자 (0x1a바이트 슬롯) | `DAT_02227f6c` |
| +0x26 | firstname_len (u8) | =실자수+1, 실자수 ≤ 13 | `DAT_02227f86` @ 193839 |
| +0x28 | firstname UTF-16LE | ≤13자 | `DAT_02227f88` |
| +0x44 | face/appearance | | `DAT_02227fa4`/`DAT_02227fa8` @ 193957~ |
| +0x50 | ability[8] (u8×8) | 포인트바이(§1.2) | `DAT_02227fb0[]` |
| +0x58 | ability_budget_base (u8) | 시작 예산 | `DAT_02227fb8` |

- **성별/진영/출신은 라디오 버튼**이라 폼이 열거값만 낸다. 진영 2종(power 2/3 = 제국/동맹 계열), 출신 blood는 **진영에 따라 선택지가 다르다**(제국계 4택, 동맹계 2택; 코드값 0~4).
- 이름은 성/이름 각각 **UCS-2 ≤13자**. 사이저 `FUN_00405720`가 `[구조체+0xb]`(=lastname_len)을 0xd와 비교해 초과 시 별도 경로 → 실질 상한 13. wire 총길이 = base + Σ(이름길이×2).

### 1.2 능력치 예산(포인트바이) — 확신도: 中 (메커니즘 확정, 숫자는 런타임/매뉴얼)

`FUN_00597380`/능력 페이지가 남은 예산을 계산:
```
DAT_02227f58(남은포인트) = DAT_02227fb8(시작예산) − Σ_i (allocated[i] − baseline[i])
   allocated = DAT_02227fb0[8]  (구조체+0x50)
   baseline  = DAT_02228480[8]  (초기 능력 기준값, 런타임 세팅)
```
- **8개 능력 스탯**을 공유 예산 풀에서 배분. `DAT_02227f58 ≥ 0` 유지가 폼 게이트. 시작예산 `DAT_02227fb8`·기준값 `DAT_02228480[]`은 계정/템플릿에서 런타임 주입되므로 **정확한 총합 숫자는 매뉴얼(docs/reference/*.pdf) 또는 런타임 덤프로 확정** — 서버는 어차피 권위 재검증하므로 게이트엔 무관.

### 1.3 0x1008 wire 바디 (실캡처 vs 메모리 구조체)

기존 wire 문서([[logh7-custom-character-creation-re]] §3.1)의 **실캡처 packed 좌표**(power@0x05, blood@0x06, sex@0x07, 이후 packed 이름)가 **서버 파서의 정본**이다. 위 §1.1은 **인메모리 구조체**(직렬화 전) 좌표로, 필드 의미·값범위의 교차근거다. 직렬화 `FUN_00405ea0`가 고정 구조체를 가변 packed wire로 압축(account_token 등 드롭). 서버는 실캡처 packed 레이아웃을 따르되 위 값 범위로 재검증.

## 2. 생성 커밋 → 로스터 반영 → 잠금해제 시퀀스 — 확신도: 高

```
로비 item1 클릭(정본 enable) → 시퀀서 state 0x19 → DAT_02217398=0x41, state→0x2d
 └ 0x2d~0x2f 세션/후보 선택 → DAT_0221670c=0x30
 └ 0x30 LB2SS(0x2009,u16 세션id) → 0x31 → 0x32 CONNECT_SS → 0x34 CERT_SS → 0x35 CERT_SS_OK
    → state = DAT_02217398 = 0x41 → default(0x3f<state<0x6a) → 위저드 FUN_00594f20
 └ 위저드 0x41 FUN_00595d90 → 0x0203 SelfInfo 송신 → 0x0204 응답(계정 슬롯 상태)
 └ 0x54 구조체 zero-init → 0x55~0x61 폼 입력 → FUN_00595de0(4) → ★0x1008 송신★
 └ 0x1008 OK(128B packed) → 위저드 성공 이벤트 0x16 → 로비 복귀
 └ 로비 재진입 시 0x2003 재요청 → 0x2004 body[0]=count++ → 캐릭터 패널 잠금 해제
```

- **캐릭터 id 노출 경로**: 0x1008 OK 바디엔 서버-부여 id가 **없다**(기존 §3.2 재확인). id는 이후 **0x2004(로스터 카드) 및 월드 로스터 트랜잭션 0x1200/0x120f/0x1201**로만 노출.
- 계정 상한: **entry_character ≤ 5, extension_character ≤ 2** (`Input_InformationAccount` 캡). 서버 권위 재검증.

## 3. 캐릭터 "선택" → 월드 진입 — 확신도: 高 (핵심 재정의)

### 3.1 개별 캐릭터 선택 메시지는 **존재하지 않는다**

정본 클라의 로비→월드 전 메시지 바디를 전수 확인한 결과, **선택된 캐릭터 id를 싣는 메시지가 없다**:
- **0x0205 SSGameLogin**: `FUN_004b76a0`가 `FUN_004b78a0(1,3,0)` — **바디 NULL**.
- **0x2009 LB2SS**: `FUN_0051bea0`가 `[u16 세션id]`(`DAT_02216714`)만 — 캐릭터 id 아님.
- **0x0203 SelfInfo**: 코릴레이터 kind 4 콜사이트는 **생성 위저드(FUN_00595d90) 단 하나** — 월드 선택 경로엔 없음.
- **0x2004**: 1756B(0x1b7 dword) 통째 복사(`FUN_0051be80` → `DAT_02216c88`), body[0]=count.

**∴ "게임 시작(item0)"은 개별 캐릭터를 고르는 게 아니라 계정 전체로 월드에 진입한다.** 서버는 계정에 묶인 캐릭터 로스터를 **0x1200/0x120f/0x1201 트랜잭션 + 0x0204**로 통째 밀어준다(은하영웅전설=계정이 함대/제독단을 지휘하는 그랜드전략 구조이므로 단일 캐릭터 선택 개념이 없다).

### 3.2 게임시작(item0) 시퀀스

```
로비 item0 클릭 → state 0x18:
   FUN_0051f1c0(...)      ; 게임시작 패널(페이지 0x35) 진입버튼 enable 게이트(§3.3)
   *(seq+4)=0x29; DAT_02217398=0x3d; DAT_02216c38=0x30
 └ 0x29~0x2c SS 접속 대기 → state=DAT_02216c38=0x30
 └ 0x30 LB2SS → 0x32 CONNECT_SS → 0x34 CERT_SS → 0x35 CERT_SS_OK → state=DAT_02217398=0x3d
 └ 0x3d~0x3f 풀스크린 페이드 → state 0x3f에서 FUN_004f31e0(0); return 1  ← 로비씬 종료·월드씬 핸드오프
월드씬:
 └ 0x0200 SSLogin(kind1) → 0x0205 SSGameLogin(kind3, 바디없음) → 0x0206 OK(게이트 0x35837e)
 └ 0x0203/0x0204 SelfInfo → 0x1200/0x120f/0x1201 로스터 → 0x0322/0x0323 월드레코드 → 전략맵
```

### 3.3 게임시작 패널 게이트 `FUN_0051f1c0` — 서버가 맞춰야 할 0x2004 필드

패널(페이지 0x35)의 진입버튼 2·3을 **각 캐릭터의 엔트리상태 필드**로 enable/disable한다. 필드는 0x2004 바디에서 복사된 `DAT_02216c88` 영역:
- 캐릭터0: `DAT_02216c8c._2_1_`(=body+6) ∈ {1,2} **AND** `DAT_02216f28`(body+0x2a0)==2 **AND** `DAT_02216f29`(body+0x2a1)!=0 → 버튼2 enable
- 캐릭터1: `DAT_02216ffa`(body+0x372) ∈ {1,2} **AND** `DAT_02217294`==2 **AND** `DAT_02217295`!=0 → 버튼3 enable
- (`DAT_02216c88`==0 이면 둘 다 disable)

**서버 요구:** 0x2004 로스터 바디에 각 캐릭터의 이 상태 바이트들을 유효값으로 채워야 게임시작 진입버튼이 켜진다. count≥1만으로는 부족 — 캐릭터 레코드의 엔트리상태(≈body+6, +0x2a0, +0x2a1 및 캐릭터 스트라이드 ~0x36e)까지 정합해야 함. 정확한 스트라이드/필드맵은 라이브 프로브 1건 권장.

## 4. 캐릭터 상한·삭제(item3/0x2008) — 확신도: 高

- **상한:** entry ≤5, extension ≤2 (§2).
- **삭제는 정본에서 이중으로 죽어 있다:**
  1. 로비 메뉴 enable 배열(state 0x16) = **`{1,1,0,0,0,1,0,1}`** → **item3(削除)=0 정적 disable** (§5로 정본 디컴파일 삼중확인).
  2. 삭제 클릭(state 0x1b)은 `DAT_02217398=0x39`로 라우팅하는데 **시퀀서 FUN_0051a370에 case 0x39 핸들러가 아예 없다**(존재 케이스: …0x35, 0x3d, 0x3e, 0x3f, 0x6a…). 0x39<0x40이라 위저드로도 안 감 → dead-end.
- 즉 item3 enable 패치를 해도 삭제는 동작하지 않는다(핸들러 부재). 삭제(0x2008)는 **코릴레이터 테이블에도 없음** — 이 빌드엔 삭제 송신 콜사이트 자체가 없다.
- 서버 codec의 0x2008 디코더(`logh7-character-codec.mjs:264`)는 형식상 존재하나 **정본 클라가 트리거하지 못하므로 라이프사이클에서 도달 불가**. (삭제 기능을 살리려면 EXE 패치로 state 0x39 핸들러 신설 필요 — 범위 밖.)

## 5. 모든 생성 변형 + 로비 메뉴 정본 확정 — 확신도: 高

정본 EXE **직접 디컴파일**(`FUN_0051a370` case 0x16)로 enable 배열을 삼중확인:
```c
local_154 = 0x00000101;  // bytes[0..3] = 01 01 00 00
local_150 = 0; local_14f = 1; local_14e = 0; local_14d = 1;  // bytes[4..7]
→ enable[8] = {1,1,0,0,0,1,0,1}
```
| item | 라벨 | enable | 생성코드 | 정본 상태 |
|---|---|---|---|---|
| 0 | ゲーム開始 | ✅ | — | 계정 전체 월드 진입(§3) |
| **1** | **新キャラクターの作成** | ✅ | **0x1008** | **유일한 라이브 생성 경로** |
| 2 | オリジナルキャラクター抽選 | ❌ | 0x1006 | 정적 disable (패치 시 동작 — 위저드 state 0x40 존재) |
| 3 | キャラクター削除 | ❌ | 0x2008 | 정적 disable + 핸들러 부재 = 이중 사망(§4) |
| 4 | セッションの変更 | ❌ | — | 정적 disable |
| 5 | 環境設定 | ✅ | — | 로컬 UI |
| 6 | クレジット(추정) | ❌ | — | disable |
| 7 | ゲーム終了 | ✅ | — | `FUN_0054eed0` |

> **[[logh7-m2-character-creation-flow]] §2 정정(삼중확인):** 그 문서의 `{1,0,1,0,1,0,0,0}`(item2 enable, item1 disable)은 **오류**다. 정본 디컴파일은 `{1,1,0,0,0,1,0,1}` — **item1(커스텀 생성)이 켜져 있고 item2(추첨)가 꺼져 있다.** 따라서 "빈 계정 첫 캐릭터 = 추첨(0x1006)" 서술은 정본과 어긋난다. **정본에서 빈 계정 첫 캐릭터의 라이브 경로는 item1 커스텀 생성(0x1008)**이다(0x1006 추첨은 item2 패치 시에만).

**생성 변형 총 3종(코릴레이터 확정):** 0x1006 추첨 / 0x1007 확장 / 0x1008 커스텀. 그 외 생성 경로 없음. 0x1007은 셀렉터(kind 0x0d)만 있고 로비 콜사이트가 없어 이 빌드에서 미사용(기존 [[logh7-m2-character-creation-flow]] §4.4와 일치).
