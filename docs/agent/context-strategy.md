# Context Strategy

> **TL;DR** 컨텍스트 선택은 작업 유형 → 필요 파일 → 필요 부분 순서 (전수 탐색 금지). 서브에이전트는 자기완결적 브리프 + 모델 명시로 수고 없이 작동. 문서는 TL;DR 우선 로딩, 신선도는 원천 변경 시 강제 갱신(hook).

컨텍스트는 유한하다. 핵심 정보가 묻히기 전에 선별·요약·리셋한다.

## 컨텍스트 유형 3종

| 유형 | 언제 | 우선순위 | 한도 | 배제 기준 | 함께 쓸 검증 | 이 저장소에서 실사용 |
|---|---|---|---|---|---|---|
| 코드 | 위치·호출경로·영향범위 질문 | `.codegraph` 먼저 → rg 확인 → 필요한 파일의 필요한 부분만 Read | 전체 파일 통독 금지, 대형 파일은 심볼·범위 지정 | 리셋 전(5bd249c 이전) 코드 경로 언급은 불신 | 테스트 실행·`git diff` | ✅ (codegraph + rg) |

## 작업 시작

- `.ai/task.md`와 승인 결정(`.ai/decisions.md`)만 우선 로드. 전체 저장소를 컨텍스트에 넣지 않는다.
- 관련 파일과 유사 구현만 탐색 (codegraph → rg → Read 순).

## 작업 중 — `.ai/current-state.md` 갱신 시점

계획 확정 후 / 주요 구현 완료 후 / 검증 실패 후 / 컨텍스트가 길어졌을 때 / 다른 에이전트로 넘기기 전.
(Stop 훅이 실작업 턴의 갱신을 강제한다.)

## 컨텍스트 리셋 권장 조건

- 작업 성격 전환 (서버 구현 → RE, 기능 구현 → 장애 대응 등)
- 확정된 규칙을 반복 누락하거나 이전 결정과 모순되는 행동이 나타남
- 관련 없는 로그가 컨텍스트 대부분을 차지
- 리셋 전 반드시 Claude `/checkpoint` 또는 Codex `logh7-checkpoint` (`.ai/current-state.md`·`handoff.md` 갱신)

## 압축 원칙

- 남긴다: 목표, 승인된 결정, 변경 파일, 검증 결과(명령+종료 코드), 실패한 접근, 다음 행동, 미해결 위험.
- 버린다: 장황한 대화, 폐기된 아이디어의 세부, 중복 로그, 무관한 파일 내용, 재현 불가능한 추측.
- `.ai/current-state.md`는 항상 최신 스냅샷 1개 — 로그 저장소가 되면 실패다.

## 컨텍스트 패키징 규칙

서브에이전트 브리프는 자기완결적이어야 한다. 다음은 포함:
- 작업 계약의 핵심 (3~5줄): 목표, 왜 중요한가, 성공 기준
- 대상 파일 목록 (절대 경로, 변경 범위)
- 검증 기준 (실행할 명령, 기대 출력, 종료 코드)
- 보고 형식 (보고서 타입, 최대 길이, 필수 포함 항목)
- 금지 사항 (절대 하지 말 것: git commit, 파일 읽기, 외부 호출)

다음은 배제:
- 전체 파일 덤프 (필요한 부분만 명시)
- 무관한 이력 (과거 시도, 이전 PR, 폐기된 계획)
- .ai/ 디렉토리의 모든 파일 (계약만 요약)
- 하이퍼링크 없이 참고 문서 다섯 개 이상

**모델 명시**: 서브에이전트 호출 시 `model=opus|sonnet|haiku` 지정. 메인 세션(Advisor)은 Fable 전용 — 긴 읽기·전수 탐색·구현은 위임, 결론만 받는다.

## 토큰 예산 규칙

**문서 로딩 순서**:
1. TL;DR 블록부터 읽기 (3~5줄로 의도·현황·다음 판정 파악)
2. 라우팅 표 확인 (현재 작업 유형과 매칭되는 문서만)
3. 필요한 섹션만 정독 (전체 읽기 금지)

**카드화**: 현행(-current) 문서와 참고 데이터는 핵심을 `.ai/key-facts.md` (≤40줄)로 카드화하고 매 턴 재주입한다. (NIAH 1중: 축약본 집중)

**브리프 원칙**: 개별 브리프는 자기완결적(standalone reading context 없이도 이해 가능)이어야 한다. 같은 내용을 두 개 이상의 문서에 쓰지 말 것.

## NIAH (Needle In A Haystack) 4중 방어

원본 문서가 맞는지 검증하는 층 4개:

**①키팩트 카드**: `.ai/key-facts.md` (≤40줄)
- 모든 현행(-current) 문서와 게이트 상태의 축약본
- 매 턴 맞춤 주입(context 낭비 최소화)

**②자동 재주입 훅**: `.claude/hooks/inject-key-facts.sh`와 `.codex/hooks/inject-key-facts.sh` (fail-open)
- 파일 변경 감지 시 자동으로 카드 콘텐츠를 context에 덧붙인다
- 훅 실패는 무시하고 작업 진행(fail-open 정책)

**③신선도 게이트**: stop-doc-gate 확장
- 파일을 변경하면 관련 카드를 강제 갱신(hook 강제, 훅 없이는 끝나지 않음)
- 예: server/src/ 변경 → logh7-roadmap-current.md M4 섹션 동기 갱신 강제
- Codex 상태는 `.codex/state/<session-hash>/`로 격리한다. 프로젝트 훅이 미신뢰이거나 부재한 환경에서는 같은 문서 게이트를 수동으로 수행하고 라이브 자동화 미확인을 보고한다.

**④회수 probe 고정**: 맥락 검증용 고정 질문
- "현재 포트는?" → 47900
- "P0 게이트는?" → 실행 환경별 client runtime·정본 EXE 계보·evidence 복구
- "리셋 커밋은?" → 5bd249c

따라서 NIAH 구멍은 모든 4개 층이 동시에 실패해야 발생한다.
